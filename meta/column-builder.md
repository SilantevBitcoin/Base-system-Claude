# Сборщик колонки направления — бриф

> **Что это.** Инструкция для отдельного окна Claude Code: собрать одну **колонку направления**
> (фронт / бэк / БД / AI / …) для системы разработки — инструменты по стадиям + роль-агент + файл-привязка.
> Окно делает тяжёлую работу и **пишет файлы само**; оркестратору возвращает компактный манифест.
>
> **Как пускать.** Новому окну: «читай `meta/column-builder.md` (в каталоге системы), собери колонку **<направление>**».
>
> **Читать вместе:** `spravochniki/dev-coding-process-kb.md` (спина стадий 0–6) · `meta/dev-system-sources-catalog.md`
> (источники рынка) · репо `github.com/cdeust/Cortex` → `.claude/agents/` (шаблоны роль-агентов).
>
> **Scope:** одна колонка за раз. Метод обкатан на **всех 6 колонках** (фронт · TS/Node · Python · БД · AI · бэк) — система собрана; этот бриф для добавления НОВОЙ колонки.

---

## 0. Модель и роли

- **Сетка:** строки = стадии спины (0–6, общие), столбцы = направления. Ты строишь ОДНУ колонку (один столбец).
- **Ты (окно) = сборщик:** исследуешь рынок, чистишь, кладёшь результат в **staging** (`staging/<col>-column/` — рабочая папка вне репо) — **НЕ** пишешь в `~/.claude/` напрямую. Оркестратору возвращаешь **манифест** (§6), НЕ контент скиллов/playbook’ов — иначе забьёшь его контекст.
- **Оркестратор (другое окно) = верификатор/интегратор:** проверяет манифест (§7), после **ревью владельца** вносит из staging в `~/.claude/`.
- Целевое место всего качаемого — глобально `~/.claude/` (туда вносит оркестратор после ревью). Скиллы и роль-агенты — на **английском**.

---

## 1. Процедура (7 шагов)

1. **Ячейки.** Назови, какие стадии спины требуют инструмента именно под это направление и что закрывают (список capability-зон: напр. для фронта — арх/UI/a11y/perf/тест/debug).
2. **Инвентарь.** Что уже установлено под направление (доступные скиллы/агенты сессии). Не предлагать дубли — искать «лучше» или «чего нет».
3. **Разведка рынка.** Кандидаты в мега-коллекциях каталога + web, по ячейкам. Учитывай уроки §3.
4. **Рубрика (4 критерия).** Покрытие · согласие с дисциплиной (слой/контракт/7-запретов/evidence-based/no-clever-code) · глубина-актуальность · цена контекста (чистый вырезаемый кусок vs раздутый мусор/стаб).
5. **Вердикт на ячейку:** взять / заменить / добавить / выкинуть. Пробел, который рынок Python-/направление-скиллом НЕ закрывает → впиши **гейт-инструкцией** в binding (как `ruff`/`mypy`/`bandit` для Python), не выдумывай скилл.
6. **Extract-clean (на каждый «взять»).** Выкачай реальный файл → определи real-vs-stub → **security-глянец** исполняемых скриптов → вырежи мусор → положи чистое в staging-папку колонки. Split-arch (тонкий `SKILL.md` + `resources/implementation-playbook.md`) — playbook копируй **вербатим**.
7. **Роль-агент.** Собери `<направление>-engineer` из Cortex-шаблона (§5).

---

## 2. Куда писать (в staging; оркестратор внесёт в `~/.claude/` после ревью)

Собираешь в `staging/<col>-column/` (создай каталог, если его нет — это рабочая папка вне репо), повторяя целевую структуру `~/.claude/`:
- **Скилл:** `…/<col>-column/skills/<name>/SKILL.md` (+ `scripts/`, + `resources/` при split-arch).
- **Роль-агент:** `…/<col>-column/agents/<направление>-engineer.md`.
- **Привязка:** `…/<col>-column/rules/<направление>.md` — таблица «стадия → инструмент» + ревью-рубрика/гейты + указатель на роль-агента + (опц.) «Примечание по применению» (язык-reference, границы scope). **Только финальные решения — что взяли и как используем. НЕ писать «Не закрыто»/«рассматривали»/донор/staging-ссылки/«добор при нужде»/«отклонён»: в binding это мусор. Что НЕ взяли → в манифест оркестратору (§6), не в систему.**

Целевые пути (куда оркестратор положит после ревью): `~/.claude/{skills,agents,rules}/`.

## 4. Анти-мусор (жёстко)

- **Стаб** = generic-шаблон «Clarify goals / Apply best practices / open implementation-playbook.md», пустые секции, битые ссылки на несуществующие `resources/*` → **ОТКЛОНЯТЬ**, не спасать.
- **Security-глянец** исполняемых скриптов ДО глобального install: нет сети наружу / `exec`/`eval`/`os.system` / чтения секретов / записи вне переданного пути / скачать-и-запустить. Stdlib-only + read-only = безопасно. Опасное — не ставить.
- **Резать:** repo-обвязку (`README`, `plugin.json`, `settings.json`), Kiro/Cursor `metadata:`-фронтматтер, маркетинг, межагентный JSON-протокол (VoltAgent/lst97 `requesting_agent`), дублирующие `## Resources`/`## Limitations`-водичку.
- **Не ссылаться** на чужой файл с пометкой «возьми только часть» — вырезать чистый кусок к себе.
- **Только нужное (YAGNI).** Что выкинул — назвать в манифесте.
- **Финальные решения, не история выбора.** В `rules/<dir>.md` идёт только: что взяли + как используем + гейты + границы scope. Отклонённые кандидаты, доноры, staging-ссылки, «добор при нужде», «вернуться к рынку», «кандидат не активен» — в binding мусор; их место (если вообще нужно) — манифест оркестратору, не система.


## 7. Верификатору (оркестратору) — лёгкая приёмка, без перечитывания тяжёлого

1. **grep роль-агента** на Cortex-мусор: `MEMORY_AGENT_ID|cortex:recall|memory-tool|scope-registry|rules/coding-standards|Lamport|Curie|Dijkstra|Liskov|Feynman|Knuth|ai-architect|/memories/` → ожидать **0** (голый `coding-standards` не флагать — это наш TS-скилл).
2. **Spot-check 1–2 скилла:** файл на месте, контент реальный (не стаб-шаблон).
3. **Если есть скрипты** — прогнать один (`--help` / известный вход): работает + подтвердить безопасность (read-only, stdlib).
4. **Binding:** стадии размечены, пробелы вписаны гейтами, роль-агент указан; **нет «Не закрыто»/доноров/отклонённых кандидатов** — только финальные решения.
5. **Харнесс листит** новые скиллы (значит распарсились валидно).
6. ОК → **внести в систему:** обновить память `dev-system-realization-stage` + индекс `MEMORY.md`.

## Итог в одну строку

**Окно: ячейки → инвентарь → разведка (каталог+web) → рубрика-4 → extract-clean (real-vs-stub + security + резать мусор) → install глобально в `~/.claude/` (англ.) → роль-агент из Cortex-шаблона → манифест. Оркестратор: grep на чистоту + spot-check + прогон скрипта → внести в систему.**
