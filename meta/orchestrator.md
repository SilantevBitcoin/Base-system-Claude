# Оркестратор — рунбук роли (верификация + интеграция)

> **Что это.** Инструкция для свежего сеанса Claude Code, который играет роль **оркестратора**: принимает
> собранную окном колонку из staging, проверяет, вносит в `~/.claude/`, обновляет память — и закрывается.
> Пара к `column-builder.md` (сторона сборщика).
>
> **Как пускать:** новому сеансу **в корне системы** (память системы живёт в памяти ЭТОГО
> проекта — `~/.claude/projects/<slug>/memory/`) — «читай `meta/orchestrator.md` +
> память проекта, прими колонку `<X>`».
>
> **Статус:** все 6 колонок (фронт · TS/Node · Python · БД · AI · бэк) собраны; метод обкатан полностью. Рунбук — для приёмки НОВОЙ колонки.

---

## 0. Принцип (почему так)

- **Окна строят** (эфемерно) → **оркестратор проверяет+вносит** (тоже эфемерно). Состояние живёт в **ПАМЯТИ**, не в контексте сеанса.
- **Оркестратор НИКОГДА не строит и не транскрибирует контент** — это забивает контекст (главная прошлая ошибка). Только: дать верификацию субагенту → `cp` → обновить память.
- Один сеанс = одна задача → обновил память → закрылся. Не марафонь.

## 1. Старт сеанса — загрузить состояние

Прочитай **только**: `MEMORY.md` (индекс) + `dev-system-realization-stage` (главный статус) + нужные `[[…]]`.
Это «сейв-файл»: что собрано, какие колонки уже в `~/.claude/`, какие решения приняты, что осталось.

---

## 2. Задача «принять колонку из staging»

Вход: окно отдало `staging/<X>-column/` (`skills/`, `agents/`, `rules/`, `scout-*`) + манифест.

1. Прочитай **манифест** окна (компактный — НЕ контент скиллов).
2. **Верификацию отдай субагенту** (он читает тяжёлое, ты — нет). Промт — §4. Вернёт: вердикт · grep-улики · safety скриптов · бизнес-привязки · теги.
3. Вердикт **ПРОШЁЛ** → интегрируй (§3). Применить мелкие фиксы из решений (отложить скилл / поправить пример / добавить tool агенту).
4. Проверь install: харнесс листит новые скиллы; есть скрипт — прогони раз (ожидаемый безопасный выход).
5. Обнови память (§5). Закрой сеанс.

---

## 3. Интеграция — механика и грабли

- `cp -r` staged `skills/`/`agents/`/`rules/` в `~/.claude/{skills,agents,rules}`. **Отложенные скиллы — НЕ копировать** (помечены в манифесте/binding; в `~/.claude` их быть не должно).
- **Деструктивное (rm / перезапись вне staging) — только спросив владельца:** назови конкретный файл и
  причину, дождись явного «да» в текущей сессии (сессия может быть в bypass — система за тебя НЕ
  спросит; см. `protect-dev-system`). Дефолт — нужное **исключай не-копированием**, это дешевле удаления.
- **`Edit` требует `Read` целевого пути.** `cp`-нул файл и надо поправить → правь **staging-копию и re-`cp`**, ИЛИ `Read` установленной копии перед `Edit`. (Один источник правды — staging.)
- **Bash проще:** `cp` без backtick / `||` / кириллицы в одной строке — спецсимволы могут сорвать permission. Проверки делай **Grep-тулом**, не bash-grep'ом.
- Деферред-скилл (рынок не дал / решили не брать) → ячейку закрывай гейт-инструкцией или активным инструментом; **НЕ ссылайся на него в binding и НЕ создавай «Не закрыто»**. Что не взяли фиксируется в памяти (статус сборки), не в binding — там только финальные решения.

## 4. Промт верификатору-субагенту (копируй)

```
Проверь собранную колонку в staging/<X>-column/. Ты читаешь тяжёлые файлы;
оркестратору возвращаешь ТОЛЬКО компактный вердикт (не контент). Проверь и верни:

1. Cortex-мусор в agents/ — grep на
   MEMORY_AGENT_ID|cortex:recall|memory-tool|scope-registry|rules/coding-standards|/memories/|agent_topic|memory_scope|ai-architect|Lamport|Dijkstra|Curie|Liskov|Feynman|Knuth
   → ожидать 0; перечисли хиты. (Именно `rules/coding-standards` — ссылка на Cortex-файл правил;
   голый `coding-standards` НЕ флагать: это наш легитимный TS-скилл.)
2. Тег-дефект — grep agents/ на бар </output> (должно быть </output-format>) → 0.
3. Стабы в skills/ — grep на «Clarify goals» / «Apply best practices and patterns relevant» /
   «implementation-playbook» без реального контента рядом → назови подозрительные SKILL.md.
4. Скрипты (*.py/*.sh) — прочитай каждый. Безопасен ли глобально: только stdlib/read-only;
   нет сети/exec/eval/чтения секретов/записи вне cwd? Вердикт по каждому.
5. Бизнес-привязки — есть ли SaaS/CRM/конкретные продукты в артефактах (должно быть универсально)?
6. Frontmatter агентов — name/description(«Use PROACTIVELY when…»)/model/tools на месте, английский?
7. Манифест vs факт — заявленное окном совпадает с содержимым папок?

Верни СТРОГО: ПРОШЁЛ / ПРОБЛЕМЫ + список проблем + список «ставить / отложить» + safety скриптов.
БЕЗ контента файлов.
```

- Пиши **коротко** (статус, не контент). Конвертируй относительные даты в абсолютные.

## 7. Полномочия оркестратора

Курирует (отложить скилл / добавить tool агенту / поправить небезопасный пример / исправить языковой дефолт) — это **делегировано владельцу**. Помечай решение в отчёте; всё **обратимо** («отложить/не ставить» безопаснее «поставить-потом-удалять»; удаление — только после явного «да» владельца на конкретный файл). Спорные/стратегические развилки — владельцу через короткий вопрос.

**Свежий сеанс: читай память → читай манифест окна → верификацию в субагента (вердикт+grep+safety) → `cp` в `~/.claude` (отложенное не бери) → обнови память → закройся.** Тяжёлый контент в свой контекст не тащи.
