# Этап 7 — Сопровождение (ревью · финал · развитие · ремонт)

> Ты здесь после того, как код написан, ИЛИ когда работаешь с уже готовым продуктом. Эти режимы
> **переиспользуют** те же справочники — отдельной системы нет, есть разные точки входа.

## Ревью и финал (завершение работы)

- `/code-review` (баги + чистка диффа; `ultra` = глубокий облачный для важного)
- `/security-review` — **безопасность, всегда для серьёзных задач** (auth/деньги/данные)
- `/simplify` — упрощение/качество
- → `superpowers:finishing-a-development-branch` → мерж/PR

Правила ревью — [`../spravochniki/dev-workflow-kb.md`](../spravochniki/dev-workflow-kb.md) §5.

## Режимы входа

| Пользователь говорит… | Что делаешь |
|---|---|
| **«новая фича»** | Крупная → назад в [`../0-ideya/`](../0-ideya/README.md) / [`../1-plan/`](../1-plan/README.md) (мини idea/PRD). Мелкая → сразу спина [`../2-realizaciya/`](../2-realizaciya/README.md), ветка «новый код в проекте» |
| **«сломалось, почини»** | Спина [`../2-realizaciya/`](../2-realizaciya/README.md), ветка «правка готового», акцент на стадию **4 Отладить** (репро → корень → фикс в источнике); скилл `systematic-debugging` |
| **«как тут устроено»** | Читай `docs/` проекта: `product.md` (что/зачем), `tech-stack.md` (стек), `decisions/` (почему так), `knowledge.md` (грабли/паттерны) |

## Принцип

Ремонт и развитие НЕ требуют отдельных документов — они входят в спину 0–6 с разными акцентами и
читают конвенцию `docs/` проекта. Контент один, точки входа разные.

Назад в оглавление: [`../CLAUDE.md`](../CLAUDE.md)
