# Система разработки — оглавление

> **Что это.** Универсальный конвейер «мысль → продукт → сопровождение» для работы с Claude.
> Один хребет, одна конвенция, ноль дублей. Подходит любому проекту (веб, автоматизация, CRM, AI).
>
> **Новый агент — читай только этот файл.** Он маршрутизирует: по намерению пользователя ты идёшь
> в ОДИН нужный справочник, а не грузишь всё. Контекст остаётся чистым.
>
> ⛔ **Защита системы.** Эти файлы (этот репозиторий + runtime `~/.claude/{agents,rules,skills,scripts}`) Claude
> НЕ меняет по своей инициативе — только по прямой конкретной просьбе владельца. Чтение свободно.
> Подробно — `~/.claude/rules/protect-dev-system.md`.

## С чего начать — вход по намерению

Пользователь говорит, что делает → ты открываешь нужную папку этапа (её `README.md` = тонкий маршрут
с шагами и ссылками). Контент лежит в `spravochniki/` — маршрут зовёт нужный.

| Пользователь говорит… | Раздел | Открой |
|---|---|---|
| «есть сырая мысль», «стоит ли это делать», «помоги сформировать идею» | идея | [`0-ideya/`](0-ideya/README.md) |
| «делаем PRD», «нужен дизайн / ADR», «составь план», «ведём новый проект» | план | [`1-plan/`](1-plan/README.md) |
| «кодим фичу», «пишем проект», «реализуем по плану» | реализация | [`2-realizaciya/`](2-realizaciya/README.md) |
| «сломалось, почини», «развиваем готовый продукт», «как тут устроено» | сопровождение | [`3-soprovozhdenie/`](3-soprovozhdenie/README.md) |
| «расширить саму систему» (новая колонка-направление, новый справочник) | мета | [`meta/`](meta/README.md) |

**Если идея уже ясная** — пропускай раздел «идея», начинай с `1-plan/`.
**Не каждый проект проходит всё** — мелкая фича: `1-plan` → `2-realizaciya` → финал (облегчённо). Демо/POC: облегчённый PRD.

## Карта системы (что где лежит)

```
Base/
├── CLAUDE.md          ← ты здесь (оглавление, вход)
│
├── 0-ideya/           ← раздел «идея» (этап 0 процесса)
├── 1-plan/            ← раздел «план» (этапы 1–4: PRD/дизайн/план)
├── 2-realizaciya/     ← раздел «реализация» (этапы 5–6: код + доки по ходу)
├── 3-soprovozhdenie/  ← раздел «сопровождение» (этап 7: ревью/финал + развитие/ремонт)
│       ↑ это ТОНКИЕ маршруты: что делать + куда идти. Контента нет.
│
├── spravochniki/      ← КОНТЕНТ по темам (1 источник на тему, ноль дублей):
│   ├── idea-kb.md              формирование идеи (этап 0)
│   ├── dev-workflow-kb.md      маршрут процесса + конвенция docs/ + шаблоны PRD/ADR
│   ├── saas-stack-kb.md        выбор стека / архитектуры / build-vs-buy (дизайн)
│   └── dev-coding-process-kb.md  спина «как кодим» — стадии 0–6
│
├── meta/              ← как система устроена и РАСШИРЯЕТСЯ:
│   ├── column-builder.md            сборка новой колонки-направления
│   ├── orchestrator.md              приёмка/интеграция колонки
│   └── dev-system-sources-catalog.md  каталог источников рынка
│
└── runtime-copy/      ← «РУКИ» системы — копия для установки в ~/.claude/:
    ├── agents/         7 роль-агентов
    ├── rules/          привязки направлений + дисциплина + поведение
    ├── skills/         66 скиллов-инструментов
    ├── scripts/        хук-напоминалка
    └── settings.example.json  шаблон конфига Claude Code
```

**РУКИ системы РАБОТАЮТ из `~/.claude/`** (иначе Claude Code их не подхватит); в репозитории они лежат в `runtime-copy/` как копия для установки:
- `agents/` — 7 роль-агентов (`frontend-engineer`, `typescript-engineer`, `python-engineer`, `dba`,
  `data-scientist`, `mlops`, `devops-engineer`).
- `rules/<направление>.md` — **8 направлений**: 6 колонок с роль-агентом (фронт · TS/Node · Python · БД · AI · бэк) + лёгкие `marketing` · `design` (без агента, план-этап). Грузятся **лениво по словам** (диспетчер, см. ниже).
- `skills/` — скиллы-инструменты направлений (описания видны всегда, тело скилла грузится при вызове; актуальный список — `ls ~/.claude/skills`).
- **Плагины** (`~/.claude/plugins/`, отдельно от `skills/`) — внешний process-хребет `superpowers`
  (brainstorming · writing-plans · executing-plans · systematic-debugging · finishing) + прикладные
  (`document-skills` · `vercel` · `karpathy-guidelines`). Зависимость, а не наш контент: снести плагин = сломать поток этапов 1–7.

Привязки направлений (`rules/<направление>.md`) грузятся **лениво по словам в чате** — диспетчер
`dev_dispatcher.py` (хук `UserPromptSubmit`) вкладывает нужную привязку + роль-агента, когда задача её
касается; вне совпадения контекст чист (`claudeMdExcludes` в `settings.json` снимает их с авто-загрузки).
Объявить направление = выбрать таблицу → роль-агент ведёт стадии 0–6 и дёргает скиллы. Гайд диспетчера —
`meta/dev-dispatcher-guide.md`. Эта папка `Base/` — «мозг» (знание); `~/.claude/` — «руки» (runtime).

**Состояние проекта** — в памяти `~/.claude/projects/<проект>/memory/` (копится само).

## Принцип навигации (почему так)

Маршрут (этапы) **тонкий** — только шаги и ссылки. Справочники (темы) — **толстые**, по одному на тему.
Этап ссылается на справочник, не копирует его. Сквозная тема (безопасность, тесты, доки) = ОДИН
справочник, на который ссылаются разные этапы → **ноль дублей контента** (дублируются только ссылки).

**Дубль ≠ переиспользование.** Дубль (запрещён) = скопированный структурированный текст (таблицы,
списки критериев, гейты) в двух местах — он рассинхронизируется. Резюме в 1–2 строки для навигации и
переиспользование общего по **ссылке** (разные этапы/действия — фича, ремонт — зовут одну спину) —
это норма, не дубль.

Один process-хребет — `superpowers`. Второй фреймворк не подключать (это и был корень «хватания»).

## Глоссарий: три шкалы нумерации (не путать)

В системе три независимые нумерации — НЕ сшивай их по совпадению числа:
- **Этапы 0–7** — процесс проекта (идея → финал). Справочник `spravochniki/dev-workflow-kb.md`.
- **Стадии 0–6** — как кодим (разворот этапа «Код» = этапа 5). Справочник `spravochniki/dev-coding-process-kb.md`.
- **Фазы роста 1–5** — эволюция стека по масштабу. `spravochniki/saas-stack-kb.md` §8.

«Этап 4» (План) ≠ «стадия 4» (Отладить) ≠ «фаза 4» (Redis/очереди). Всегда называй шкалу словом.

## Подключение к проекту

Система живёт здесь, в проекты НЕ копируется — проекты на неё **ссылаются**. Чтобы проект пользовался
системой: скопируй `dev-system.md` (лежит рядом) в корень проекта и подключи его (инструкция внутри
файла). Маршрут активируется автоматически в каждой сессии проекта через `@`-импорт этого оглавления.
