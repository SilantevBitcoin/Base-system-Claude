# База знаний: выбор стека, архитектуры и build-vs-buy для SaaS / AI-проектов

> **Что это.** База знаний по тому, как выбирать технологический стек, архитектуру и
> «писать самому vs взять готовое» при создании софта, SaaS и AI-продуктов. Самодостаточна —
> читается с нуля, без внешнего контекста.
>
> **Зачем.** Чтобы не объяснять всё заново каждой новой сессии ассистента и в каждом проекте.
> Дал файл → ассистент понимает варианты и принимает решения по этим правилам.
>
> **Как читать:**
> - **Ассистент (AI-копия):** читай **Часть I (Playbook)** — там правила, таблицы выбора и готовые
>   стек-рецепты. В **Часть II** ныряй за объяснением «почему», когда нужно обосновать решение
>   пользователю или он спрашивает «что это такое».
> - **Человек:** читай всё; Часть II объясняет понятия простым языком.
>
> **Статус актуальности.** Срез на **2026-06**. Принципы (TCO, переносимость, build-vs-buy,
> анти-overengineering) устаревают медленно. Конкретные названия библиотек/сервисов и их статусы
> (версии, «deprecated», цены) — проверяй перед применением.
>
> **Источники и приоритет.** Сведено из трёх источников; при расхождении приоритет у двух
> «переписок» (обучающий диалог + конспект `base-saas.md`), внешний research — дополнение. См.
> «Источники» в конце.

# ЧАСТЬ I — PLAYBOOK (для AI-копии: правила и решения)

## 0. Инструкция ассистенту: как применять этот файл

Когда пользователь описывает проект и хочет помощи с технической стороной:

1. **Сначала классифицируй проект** по таблице §3 (тип → готовый стек-рецепт). Не изобретай стек с нуля.
2. **Предложи дефолтный стек** (§2), затем адаптируй под особенности из описания.
3. **На каждый «кусок» системы примени build-vs-buy** (§5): по умолчанию — взять готовое; писать
   самому только ядро-дифференциатор и то, что иначе создаёт дорогой lock-in.
4. **Проверь по анти-паттернам** (§6): не предлагаешь ли ты overengineering (K8s/микросервисы на
   нулевой аудитории), не вешаешь ли всё на проприетарный vendor.
5. **Заложи фундамент переносимости день-1** (§7), даже если деплой на managed-платформу.
6. **Surface assumptions.** Если выбор стека зависит от факта, которого нет в описании (ожидаемая
   нагрузка, нужен ли тяжёлый AI/GPU, команда или solo, бюджет, регуляторика) — спроси, не угадывай.

Главный приоритет решений: **скорость до первого работающего MVP** (для раннего проекта) ×
**отсутствие дорогого lock-in** (всегда). Всё остальное — потом.

## 1. Правила-аксиомы (TL;DR всей базы)

1. **«Лучшего языка» нет — есть лучший под задачу.** Дефолт для большинства SaaS — TypeScript;
   для тяжёлого AI/ML — добавляется Python. (§II-A)
2. **Бери managed, чтобы быстро запуститься.** Час инженера дороже месяца managed-сервиса; на
   раннем/среднем этапе managed почти всегда дешевле по **TCO**, несмотря на больший «счёт за
   сервисы». (§II-H)
3. **Считай TCO, а не счёт за сервисы.** У «своего сервера» прямая цена низкая, скрытая (DevOps,
   аптайм 24/7, бэкапы, failover, масштабирование) — огромная. (§II-H)
4. **Единственный по-настоящему опасный lock-in — проприетарная БД/состояние.** Хостинг сменить —
   день; модель данных переписать — месяцы. (§II-J)
5. **Начинай на платформе, но с день-1 закладывай границы переносимости** — ~5% дисциплины → 80%
   лёгкости будущего переезда. Не «своя инфра на вырост», а архитектурные границы. (§II-I)
6. **Build-vs-buy по умолчанию = buy.** Пиши сам только (а) ядро-дифференциатор продукта и (б) то,
   что иначе вешает дорогой lock-in. Auth, платежи, email, БД-движок — не пиши. (§5)
7. **Монолит, не микросервисы.** Разделишь, когда заболит. Почти никто не прыгает сразу в
   K8s/микросервисы — двигаются по спектру по мере роста боли. (§6, §8)
8. **Данные — в стандартном Postgres.** Универсален, self-hostable, не лочит. Дефолт SaaS.
   Избегай Firebase/DynamoDB как ядра. (§II-J)
9. **Docker с день-1** — паспорт переносимости, даже если деплоишь на PaaS. (§7)
10. **Можно начать на Next.js и добавить Python позже** через отдельный HTTP-сервис — это норма, не
    костыль; переписывать с нуля не нужно. (§II-E)
11. **Product thinking > код.** UX, distribution, retention, монетизация важнее выбора фреймворка.
    Не полируй инфру на нулевой аудитории. (§II-G)
12. **Главный навык 2026 — проектировать системы и быстро собирать MVP с AI**, а не помнить
    синтаксис. (§II-G)

## 2. Дефолтный стек 2026 (modern AI SaaS)

Отправная точка для большинства проектов; меняй прицельно, не целиком.

| Слой | Дефолт | Заметка |
|---|---|---|
| Язык | **TypeScript** | один язык на фронт и бэк |
| Frontend | **React + Next.js** | Next.js — де-факто стандарт SaaS |
| Стили | **Tailwind CSS (v4)** + **shadcn/ui** | shadcn = компоненты копируются в проект (низкий lock-in) |
| Backend (старт) | **Next.js Route Handlers** (`app/api/.../route.ts`) | отдельный сервер не нужен на старте |
| БД | **PostgreSQL** (managed: Supabase / Neon) | настоящий Postgres = низкий lock-in |
| Доступ к БД | **Prisma** или **Drizzle** (ORM) | + тонкий свой слой-репозиторий |
| Auth | **Clerk / Supabase Auth / Auth.js** | бери с экспортом юзеров; **не Lucia** (deprecated) |
| Платежи | **Stripe** | стандарт, лочит мало |
| AI | **OpenAI / Anthropic (Claude) API** | через свой адаптер `ai.ts` |
| Vector / RAG | **pgvector** (внутри Postgres) | не плодить отдельную vector-БД раньше времени |
| Очереди / jobs | **Redis + BullMQ** (TS) / **Celery** (Python) | добавляется, когда появляются долгие задачи |
| Тяжёлый AI/ML | **Python + FastAPI** (отдельный сервис, позже) | связь по HTTP |
| Контейнер | **Docker** (день-1) | переносимость |
| Деплой | **Vercel** (фронт/MVP) → Railway/Render/Fly/Cloud Run по мере роста | |

**Solo-founder / AI MVP за вечер:** `Next.js + Supabase + Stripe + OpenAI/Claude API + Vercel`.

## 3. Готовые стек-рецепты по типу проекта

| Тип проекта | Рекомендуемый стек | Почему так |
|---|---|---|
| **MVP / валидация идеи** | Next.js + Supabase + Stripe + Vercel | максимум скорости, минимум инфры |
| **AI SaaS (чат/ассистент/RAG)** | Next.js + Supabase(pgvector) + OpenAI/Claude + Stripe; Python FastAPI добавить под тяжёлый AI | TS для продукта, Python для AI-движка |
| **Internal tool / dashboard / CRUD** | Next.js + Supabase + shadcn/ui | auth+DB+API из коробки |
| **Telegram-бот / automation** | Next.js (панель) + Python (воркеры/скрейпинг) + Postgres + Redis | Python силён в automation |
| **Highload API / realtime / websockets** | **Go** для горячего пути + Postgres + Redis; фронт всё равно Next.js | Go: concurrency, низкий RAM |
| **Enterprise / банковский уровень** | **Java/Kotlin** или **NestJS** (TS) + Postgres + K8s | зрелость, строгая архитектура, надёжность |
| **Контентная платформа / SEO-важен** | Next.js (SSR/SSG) + Postgres + CDN | SSR + SEO из коробки |
| **Ultra-fast MVP без отдельного фронта** | Python: FastAPI/Django + HTMX + Jinja + Postgres | один язык, продукт за неделю; слабее UI/масштаб команды |

Backend-фреймворк по размеру: **Route Handlers** (маленький/средний SaaS) → **FastAPI** (AI/ML
сервис) → **NestJS** (большой enterprise-бэкенд, строгая архитектура). Express — легаси, для нового
SaaS обычно не берут.

## 4. Managed vs своё — когда что (таблица решений)

Спектр контроля (ключевая модель):

```
свой физ. сервер → VPS (Hetzner/DO) → AWS EC2 → AWS managed → Vercel
◄── больше контроля, дешевле на масштабе, больше возни
              меньше возни, дороже за единицу, меньше контроля ──►
```

| Ситуация | Выбор |
|---|---|
| MVP, ранний этап, маленькая команда | **managed/PaaS** (Vercel + BaaS) |
| Счёт за managed начал кусаться, нагрузка предсказуема | **managed-контейнеры** (Cloud Run/Fargate) или **self-hosted PaaS** (Coolify/Dokku) на дешёвом VPS (Hetzner) |
| Большой стабильный объём **И** есть DevOps-ресурс **И** особые требования (регуляторика/GPU/латентность) | **VPS-кластер / K8s / своё железо** для ядра, managed для периферии |
| Нужна дешёвая постоянная нагрузка | **VPS (Hetzner/DigitalOcean)**, не AWS (AWS — тоже аренда, просто «голее») |

**Правило окупаемости своего сервера (все три сразу):** (1) большой стабильный объём, (2) есть
команда/время на DevOps, (3) особые требования. Пока managed-счёт меньше зарплаты человека, который
будет админить замену, — managed дешевле. Это часто **тысячи $/мес** managed, прежде чем своё
окупится.

> AWS vs Vercel — это **не конкуренты на одном уровне**: Vercel (PaaS) работает поверх облаков
> (включая AWS). Vercel = «квартира с мебелью», AWS = «участок + стройматериалы». См. §II-I ниже.

## 5. Build vs Buy — что брать готовым, а что писать самому

**Главное правило:** *бери готовым всё, что не является твоим конкурентным преимуществом. Пиши сам
только (а) ядро-дифференциатор продукта, или (б) то, что иначе создаёт дорогой lock-in (проприетарная
модель данных/состояние).*

> ⚠️ Авто-инструмента, который сам выдаёт build-vs-buy вердикт под твой стек, на 2026 **не
> существует** (см. §11). Это решение принимает ассистент/человек по таблице ниже + каталоги готовых
> решений.

| Категория | Брать готовым (чем) | Писать самому — когда | Lock-in / нюанс |
|---|---|---|---|
| **БД-движок** | managed Postgres: **Supabase / Neon / RDS** | ~никогда | низкий (это Postgres). Избегать Firebase/DynamoDB как ядра |
| **Auth / identity** | **Clerk / Supabase Auth / Auth.js (NextAuth) / Better-Auth** | почти никогда | **средний** → бери с экспортом юзеров, не вшивай глубоко. **Lucia — deprecated** |
| **Доступ к БД (ORM)** | **Prisma / Drizzle** | свой тонкий слой-репозиторий поверх | низкий |
| **Платежи / подписки** | **Stripe** (+ webhooks) | ~никогда | низкий |
| **Email (transactional)** | **Resend / Postmark / SES** | ~никогда | низкий |
| **Очереди / background jobs** | **Redis+BullMQ** (TS), **Celery** (Py), **Inngest / Trigger.dev** | свою логику воркера — да; брокер — нет | низкий–средний |
| **Vector search / RAG** | **pgvector** в Postgres (дефолт); **Pinecone/Weaviate** только при масштабе | ~никогда | pgvector = минимум lock-in |
| **Файлы / storage** | **Supabase Storage / S3 / Cloudflare R2** | ~никогда | низкий, если S3-совместимое |
| **Realtime / live** | **Supabase Realtime / Pusher / Ably**; свой WS позже | при особых требованиях | низкий на старте |
| **UI-компоненты** | **shadcn/ui / Radix** | свои — поверх | shadcn копируется в проект → низкий lock-in |
| **Аналитика / флаги / ошибки** | **PostHog / Sentry / feature-flag сервис** | ~никогда | низкий |
| **Деплой / хостинг** | **Vercel / Railway / Render / Fly / Cloud Run** | своё — поздно и гибридом | низкий, если Docker |
| **Бизнес-логика, доменные правила, ваш core value** | — | **ВСЕГДА сам** | это и есть продукт |

**Managed смело vs осторожно:**
- **Смело** (дешёвый lock-in): хостинг compute, managed Postgres/Redis, email, Stripe, CDN, статика,
  stateless-функции.
- **Осторожно** (думай заранее): auth/identity (бери с экспортом), всё проприетарно-stateful
  (Firebase/DynamoDB), BaaS целиком как ядро бизнес-логики.

## 6. Анти-паттерны (грабли — чего НЕ делать)

- ❌ **Преждевременный масштаб.** Kubernetes / микросервисы / distributed systems при 0 пользователей.
  → Монолит на платформе, разделишь когда заболит.
- ❌ **«Cloud-agnostic слой над всем».** Достаточно тонких адаптеров на каждый сервис. (§7)
- ❌ **Своя инфра «на вырост»** до того, как managed-счёт превысил стоимость человека-DevOps.
- ❌ **Проприетарная БД/состояние как ядро** (Firebase/DynamoDB) — самый дорогой lock-in.
- ❌ **Писать самому auth/платежи/email/БД-движок.** Это решённые задачи — бери готовое.
- ❌ **«Потом мигрирую» без границ день-1.** Если прибился к вендору намертво — «потом» = переписать всё.
- ❌ **Идеальная архитектура до проверки спроса.** Сначала MVP → спрос → масштаб.
- ❌ **Считать только «счёт за сервисы»** при сравнении managed vs своё (игнорируя TCO).
- ❌ **Express для нового SaaS** без причины — мало структуры, быстро в хаос (легаси-выбор).
- ❌ **Отдельная vector-БД**, когда хватает pgvector.

## 7. Фундамент проекта день-1 (переносимость за ~5% усилий)

**Идея одной фразой:** логика в центре, инфраструктура по краям за обёртками, данные — в стандартной БД.

```
[ роуты/входы ]  →  [ бизнес-логика (чистая) ]  →  [ адаптеры (обёртки инфры) ]
   тонкие              ядро, не знает о БД/HTTP        Postgres / Redis / Stripe / email / AI
```

**Структура (ports & adapters lite):**
```
src/
  domain/        бизнес-логика (чистая, тестируемая, без инфры)
  adapters/      db.ts, cache.ts, payments.ts, email.ts, ai.ts  — обёртки на сервисы
  api/           роуты/хендлеры (тонкие: распарсил → позвал логику → отдал ответ)
  config/        env в одном месте (12-factor), валидация env при старте
db/
  migrations/    versioned-миграции (Drizzle/Prisma) с первого дня
Dockerfile        паспорт переносимости — даже если деплоишь на PaaS
```

**7 фундаментных решений:**

| Решение | Почему |
|---|---|
| **БД = Postgres** | универсален, self-hostable, не лочит; Redis — кэш/сессии/очереди |
| **Доступ к БД через слой** (Drizzle/Prisma) | SQL не размазан; смена хостинга БД = ноль изменений в логике |
| **Миграции с день-1** | схема под версионным контролем, не «ALTER руками в проде» |
| **Docker сразу** | один контейнер крутится Vercel→Cloud Run→VPS→K8s без переписывания |
| **Конфиг через env + валидация при старте** | секреты не в коде; падай сразу при кривом env |
| **Тесты на чистую логику** | она переносима и критична — её и покрывай, а не моки инфры |
| **Stripe + переносимый auth** | Stripe лочит мало; auth — провайдер с экспортом юзеров |

**Что НЕ делать с начала:** не K8s, не микросервисы, не «слой над всем», не своя инфра на вырост.
YAGNI на инфраструктуру; дисциплина — только на данные и границы.

## 8. Эволюция стека по фазам роста

Почти никто не прыгает сразу в конец. Двигайся по мере роста боли:

```
Фаза 1:  Next.js (+ Supabase/Postgres)              ← MVP
Фаза 2:  + Stripe + OpenAI/Claude                    ← монетизация + AI
Фаза 3:  + Python AI-сервис (FastAPI, по HTTP)       ← тяжёлый AI/ML/automation
Фаза 4:  + Redis + очереди + воркеры                 ← долгие фоновые задачи
Фаза 5:  Custom infrastructure (K8s/VPS/своё железо) ← реальный масштаб + команда
```
> Это **фазы роста стека** (по масштабу), НЕ «этапы» процесса 0–7 и НЕ «стадии» кода 0–6 — глоссарий в `CLAUDE.md`.

Добавление Python на фазе 3 **не требует переписывания** фронта/БД/auth — поднимаешь отдельный
сервис и зовёшь по HTTP (§II-E).

# ЧАСТЬ II — ОБЪЯСНЕНИЯ (для человека: «что это и почему»)

## A. На каком языке делать SaaS

«Лучшего» языка нет — есть лучший под задачу. Практический срез:

- **TypeScript — универсальный дефолт.** Один язык на фронт и бэк, огромная экосистема, идеален для
  AI-assisted (vibe) coding, куча готовых SaaS-boilerplate, легко нанимать, лучший DX. Типовой стек:
  React/Next.js + Node.js/NestJS + PostgreSQL + Prisma + Clerk/Auth.js + Stripe.
- **AI SaaS = TypeScript + Python.** TS — UI/API/продуктовая логика; Python — AI/ML/scraping/
  automation (PyTorch, HuggingFace, LangChain, OpenCV, Whisper). Де-факто стандарт AI-стартапов.
- **Go — для очень большого highload.** Быстрый, мало RAM, отличный concurrency, простая деплойка,
  идеален для микросервисов/realtime/очередей. Минус: медленнее писать продукт, слабее fullstack-экосистема.
- **Java/Kotlin — enterprise/банковский уровень.** Надёжность, зрелость, масштаб. Минус: медленная
  разработка, тяжёлый DX — не для быстрого MVP.
- **Python — ultra-fast MVP без отдельного фронта.** FastAPI/Django + HTMX + Jinja + Postgres —
  продукт за неделю. Минус: слабее frontend-экосистема и масштабирование большой продуктовой команды.

Что используют популярные AI SaaS: **Next.js + TypeScript + Python + PostgreSQL + Redis + Docker** —
практически новый стандарт индустрии.

| Задача | Лучший выбор |
|---|---|
| Универсальный SaaS / solo / vibe coding | TypeScript |
| AI SaaS | TypeScript + Python |
| Highload | Go |
| Enterprise | Java/Kotlin |
| Сверхбыстрый MVP | Python |

## B. Слои современного SaaS (карта местности)

Аналогия: SaaS = ресторан. HTML — каркас здания; CSS — интерьер; JavaScript — электрика/механика;
TypeScript — правила безопасности для электрики; React — сборка помещений из модулей; Next.js —
весь ресторанный комплекс; PostgreSQL — склад и учёт; API — курьеры между системами; AI — ИИ-сотрудники.

1. **Frontend** — что видит пользователь: HTML/CSS/JS/TS → React → Next.js → Tailwind + shadcn/ui.
2. **Backend** — логика продукта: Node.js / Next.js API / Python; auth; платежи; webhooks.
3. **Database** — хранение: PostgreSQL + ORM (Prisma/Drizzle).
4. **Infrastructure** — как запускается: Docker, деплой (Vercel/Railway/Render/AWS).
5. **AI layer** — LLM API (GPT/Claude/Gemini), RAG, embeddings, vector DB.
6. **Очереди/realtime** — Redis, очереди, воркеры; WebSockets.
7. **Product logic** — UX, distribution, retention, монетизация (важнее кода).

Как соединяется: пользователь открывает сайт → React рисует UI → Next.js обрабатывает запрос → API
идёт к OpenAI → PostgreSQL сохраняет → AI возвращает результат → React обновляет экран.

## C. React vs Next.js

- **React** — это библиотека для интерфейсов (UI из компонентов-«LEGO»: `<Navbar/>`, `<Chat/>`,
  `<Pricing/>`). Сам по себе НЕ даёт backend, роутинг, auth, БД, SSR, API.
- **React-фреймворк (Next.js)** — надстройка над React, добавляет структуру, backend, роутинг, SSR,
  API-routes, оптимизацию, деплой, кеширование, server components. «С React собираешь детали — с
  Next.js строишь весь продукт».
- Почему для SaaS: компоненты переиспользуются, AI хорошо пишет React-код, огромная экосистема,
  идеально для dashboard/AI-интерфейсов/streaming.

## D. Backend: Node.js vs Next.js API vs Python

Где путаются:
- **Node.js** — среда выполнения JavaScript на сервере (позволила писать фронт и бэк на одном языке).
- **Next.js API (Route Handlers)** — backend *внутри* Next.js, работает на Node.js. Файл
  `app/api/chat/route.ts` с `export async function POST(req){...}` = endpoint. Фронт и бэк в одном
  проекте, один деплой. Отлично для SaaS/dashboard/auth/Stripe/CRUD/AI-wrappers. Хуже для тяжёлого
  AI/видео/ML/highload-compute.
- **Python** — отдельный язык/экосистема, доминирует в AI/ML/automation/scraping/data science
  (PyTorch, TensorFlow, HuggingFace, LangChain, pandas, Selenium).

Аналогия: **Next.js/Node = офис и менеджеры; Python = инженеры и фабрика.**

## E. Можно ли начать на Next.js и добавить Python позже?

**Да, и так делают в проде. Переписывать с нуля не нужно** — это одна из причин, почему стек
Next.js + TypeScript + Python стал стандартом AI SaaS.

Начинаешь с Next.js (frontend + API + Postgres), делаешь auth/dashboard/подписки/простую AI-логику.
Когда появляется тяжёлая задача (AI-агент, scraping, генерация видео, STT, CV, ML) — поднимаешь
**отдельный Python-сервис** и связываешь по HTTP:

```
Next.js:  await fetch("http://python-api/generate")
Python:   @app.post("/generate")
```

Фронт (React/Next.js), БД (Postgres), auth/платежи (Clerk/Stripe) — остаются как были. Это не
костыль, а нормальная production-архитектура: `Next.js = product layer`, `Python = AI engine`.
Вывод: **не надо заранее идеально выбирать архитектуру** — современный SaaS строится модульно.

## F. Supabase: что это, с чем комбинируют, ограничения

**Supabase = готовый backend для SaaS; грубо — «Firebase, но на PostgreSQL и open-source».** Даёт из
коробки: настоящий **PostgreSQL**, **auth** (email/пароль, Google/GitHub, magic links, OTP),
**авто-API** на таблицы, **realtime**, **storage**, **Edge Functions**. Позволяет сделать SaaS без
отдельного backend-сервера и без ручного написания auth.

`const { data } = await supabase.from("users").select("*")` вместо целого слоя Node/Express/контроллеров.

**Комбинируют с:**
- **Next.js** — фронт + красивый UI/SSR/SEO (самая популярная связка).
- **Stripe** — `Stripe webhook → Supabase` обновляет `plan=PRO`.
- **AI** — `Next.js → Supabase → OpenAI/Claude`; хранит users/chats/prompts/embeddings/AI-history/subscriptions.
- **Python** — Supabase хранит данные/auth/realtime, Python тянет AI/automation/scraping/ML.
- **pgvector** — embeddings/semantic search/RAG прямо внутри Postgres (большой плюс).

**Ограничения (важно):** Supabase заменяет *часть* инфраструктуры, **не** backend-инженерию целиком.
Сложная бизнес-логика (permissions/workflows/orchestration) выносится в Next.js-бэкенд или Python.
Плохо подходит для long-running задач (рендер видео, AI-пайплайны, heavy scraping, GPU) — для этого
нужны Python-воркеры/очереди. Не заменяет Redis/RabbitMQ/Celery/BullMQ. Остаётся vendor-зависимость
(хоть и open-source). Edge Functions — для лёгкой логики/webhooks, не полноценный backend.

**Где идеален:** MVP, AI SaaS, dashboards, internal tools, CRUD, валидация стартапа, solo founders.
**Где тесно:** Uber-подобные платформы, огромный realtime, massive AI-orchestration, enterprise ERP.
**Главное:** большинство SaaS никогда не упираются в лимиты Supabase — преждевременно думать о K8s
при 0 пользователей вредно.

> Согласование источников: ChatGPT-диалог максимально pro-Supabase для MVP; конспект `base-saas.md`
> добавляет осторожность — *managed Postgres бери смело (это настоящий Postgres), но не вешай всю
> бизнес-логику только на BaaS*; держи логику переносимой (§7). Обе позиции совместимы: Supabase как
> ускоритель — да; Supabase как единственное место всей логики — осторожно.

## G. Что ещё знать (карта тем для роста)

- **Frontend+:** Tailwind (стандарт UI), UI-библиотеки (shadcn/ui, Radix).
- **i18n / локализация:** мультиязычность как инфраструктура — UI-тексты не захардкожены, а берутся из словарей по локали; даты/числа/валюты и маршруты (`/en`, `/ru`, `/th`) разводятся по языку. Нужна, когда продукт обслуживает несколько языков (международная аудитория, мультиязычный рынок). Дефолт для Next.js App Router — **next-intl** (альтернатива — next-i18next). Закладывать с начала: ретрофит мультиязычности по готовым экранам — дорогой.
- **Backend+:** Authentication (Clerk/Auth.js), Payments (Stripe: подписки/карты/invoices/webhooks),
  Webhooks (внешний сервис уведомляет твой бэкенд).
- **Database+:** ORM (Prisma) — работа с БД как с объектами: `prisma.user.findMany()` вместо SQL.
- **Infra:** Docker (упаковка в контейнер — одинаково работает везде), Deploy (Vercel/Railway/Render/AWS).
- **AI layer:** LLM API; **RAG** (AI ищет в твоих данных — документы/PDF/база знаний); **embeddings**
  (текст → векторы для semantic search и «памяти» AI); **Vector DB** (Pinecone/Weaviate — или pgvector).
- **Очереди:** долгие задачи (видео/AI/рендер/scraping) нельзя держать в запросе → queue + worker;
  **Redis** (очереди/кэш/realtime/сессии).
- **Realtime:** WebSockets (мгновенный UI, AI-streaming, live dashboards).
- **Git/GitHub:** обязательно (история/откат/коллаборация).
- **Архитектура:** Monolith (всё в одном — лучше для MVP) vs Microservices (отдельные сервисы — для
  масштаба).
- **Product thinking:** UX, distribution, retention, монетизация — важнее самого кода.

**Главный навык 2026:** не «писать код руками наизусть», а проектировать системы, быстро собирать
MVP, правильно использовать AI-coding, понимать архитектуру, уметь дебажить сгенерированный код.

## H. Managed-сервисы vs свой сервер — считай TCO

Частая ошибка: «серьёзный SaaS → нужен свой сервер, иначе managed задушит абонентской платой» —
половина правды.

**Где верно:** на большом стабильном объёме managed дорожает (поштучная оплата может превысить фикс-
цену сервера); свой сервер = контроль и независимость; для предсказуемой постоянной нагрузки железо
выгоднее «оплаты за факт».

**Ловушка:** считать надо **TCO (полную стоимость владения)**, а не только счёт за сервисы. Прямая
цена VPS низкая ($20–100/мес), но скрытая — огромная:

| Скрыто в «своём сервере» | Цена |
|---|---|
| Настройка, мониторинг, обновления безопасности, бэкапы | твоё время / зарплата DevOps |
| Аптайм 24/7, реакция на инцидент ночью | сон и нервы |
| Надёжность (репликация, failover, SLA) | строишь сам — или теряешь клиентов |
| Масштабирование под скачок | твоя забота, не авто |

**Час инженера дороже месяца managed-сервиса.** На раннем/среднем SaaS managed почти всегда дешевле
по TCO, *несмотря* на больший счёт. Managed serverless = платишь за использование (нет трафика → ~0);
свой сервер жрёт деньги всегда. Крупные SaaS массово сидят на managed — это не «для маленьких».
Кто переезжает на своё — делает это **гибридом и поздно**.

**Окупается своё (все три сразу):** большой стабильный объём + команда/время на DevOps + особые
требования (регуляторика/данные нельзя в чужое облако/GPU/низкая латентность).

## I. AWS vs Vercel — разные уровни абстракции

**Vercel работает поверх облаков (включая AWS)** — слой удобства над инфраструктурой, не «соперник на
том же уровне».

- **Vercel = PaaS.** Готовая квартира с мебелью: `git push` и работает, планировку не меняешь. Узко и
  отполировано (фронт/Next.js/serverless). Пологий вход, ноль DevOps, дороже за единицу на масштабе.
- **AWS = IaaS+.** Участок + стройматериалы: построй что угодно, обслуживай сам. 200+ сервисов, любой
  стек, полный контроль. Крутой вход, нужен DevOps, дешевле на масштабе (если умеешь); легко словить
  сюрприз-счёт.

Нюанс: **AWS ≠ «свой сервер дешевле»** — это тоже облако (аренда), просто «голее». По-настоящему
дёшево на постоянной нагрузке — VPS у Hetzner/DigitalOcean, не AWS. Частый путь: фронт на Vercel,
бэкенд переносишь на AWS/VPS, когда упёрся в рамки.

**Варианты деплоя — весь спектр:**
```
PaaS → Serverless → Managed-контейнеры → Kubernetes → VPS → своё железо
◄── проще, дороже за единицу        дешевле на масштабе, больше возни ──►
```
1. **PaaS / фронт-облака** — Vercel, Netlify, Cloudflare Pages, Render, Railway. `git push` и забыл.
2. **Serverless (FaaS)** — AWS Lambda, Cloudflare Workers, GCF. Платишь за вызовы, near-zero при
   простое, stateless (нужно внешнее состояние — Redis/очереди).
3. **Managed-контейнеры (Docker)** — Cloud Run, Fargate/ECS, Fly.io, Railway. Любой стек, переносимость.
4. **Kubernetes** — EKS/GKE/DO K8s. Масштаб, микросервисы; высокая сложность, нужна команда.
5. **VPS (IaaS)** — Hetzner (дёшево!), DigitalOcean, Linode, EC2. Полный контроль, дёшево на постоянной нагрузке.
6. **Своё железо** — максимум контроля; капзатраты и обслуживание. Для регуляторики/особых требований.

**Гибриды:** **self-hosted PaaS** (Coolify/Dokku/CapRover — Vercel-опыт на своём дешёвом VPS);
**BaaS** (Supabase/Firebase/Appwrite); **static** (GitHub Pages, S3+CloudFront); **edge** (Cloudflare
Workers, Deno Deploy).

## J. Переносимость и lock-in (где дёшево, где дорого)

Вопрос «начать на чужой платформе и потом перейти, или закладывать своё сразу?» — ответ **посередине**:
- ❌ «Всё равно, потом мигрирую» — наивно (прибился намертво → «потом» = переписать всё).
- ❌ «Закладывать своё сразу» — over-engineering (большинство SaaS не доживает до масштаба).
- ✅ **Начинаешь на платформе, но день-1 закладываешь архитектурные границы** (не свою инфру). ~5%
  дисциплины → 80% лёгкости переезда. Как именно — см. §7.

**Где lock-in дёшев vs дорог:**

| Дёшево мигрировать (начинай смело) | Дорого (думай заранее) |
|---|---|
| Хостинг compute (Vercel→Cloud Run→VPS, если Docker/Next) | **Firebase / DynamoDB** — проприетарная модель данных, лочит намертво |
| Managed Postgres/Redis → self-host того же софта | Vendor-специфичные auth/identity, глубоко вшитые |
| CDN, статика, stateless-функции | BaaS целиком как ядро логики |

**Главное правило:** stateless (compute) мигрирует легко; состояние (БД) — вот где lock-in кусается.
**Хостинг поменять — день; модель данных переписать — месяцы.** Будь осторожен с выбором БД и не
прибивай данные к проприетарному формату.

## Итог в одну строку

**Бери managed, чтобы быстро запуститься, на дефолтном стеке (Next.js + TS + Postgres/Supabase +
Stripe + LLM API), но держи логику чистой, инфру за обёртками, данные в Postgres, всё в Docker; пиши
сам только ядро-дифференциатор; не лезь в K8s/микросервисы до реальной боли.** Единственный
по-настоящему опасный lock-in — проприетарная БД/состояние.
