# База знаний: визуальный дизайн — как создать дизайн-документ корпоративного стиля

> **Что это.** База знаний о том, как создать и поддерживать **дизайн-документ** («Brand & Design
> Core») — единый источник визуального стиля, по которому разные исполнители (дизайнер, ИИ-агент,
> подрядчик, сотрудник) делают **согласующиеся** продукты: сайт, презентацию, визитку, бланк, UI.
> Самодостаточна — читается с нуля.
>
> **Зачем.** Чтобы не объяснять стиль заново каждому исполнителю и каждой сессии ассистента. Дал
> документ → исполнитель делает в корпоративном стиле, и результаты разных рук **гармонично
> сочетаются**. Механизм этого сочетания — не «брендбук-картинка», а машиночитаемые **токены** +
> текстовый контракт «делать / НЕ делать», которые одинаково читают человек и ИИ.
>
> **Как читать:**
> - **Ассистент (AI-копия):** читай **Часть I (Playbook)** — правила, анатомия документа, модель роста,
>   процесс, анти-паттерны. В **Часть II** ныряй за объяснением «почему», когда нужно обосновать
>   человеку или он спрашивает «что это такое».
> - **Человек:** читай всё; Часть II объясняет понятия простым языком.
>
> ⛔ **Граница (важно, чтобы не было дублей).** Этот справочник — про **создание дизайн-документа**
> на этапе идея-план (его выход — артефакт в `docs/` проекта). **Производство** по этому документу
> (React/CSS-код, ревью UI, доступность, перф) живёт НЕ здесь, а в `frontend-design` /
> `web-design-guidelines` / `a11y-audit` / `web-performance` (стадии кода 2–6, см.
> `~/.claude/rules/frontend.md`). Справочник их **зовёт**, не дублирует. Документ, рождённый здесь, =
> **вход** для `frontend-design` на стадии кода.
>
> **Статус актуальности.** Срез на **2026-06**. Принципы (токены, роли цветов, уровни зрелости,
> markdown-first) устаревают медленно. Конкретные инструменты (Figma MCP, генераторы картинок,
> версии библиотек) — проверяй перед применением.
>
> **Источники и приоритет.** Сведено из четырёх источников: мировые практики дизайн-систем
> (research-диалог), обучающий урок (декомпозиция стиля + AI-пайплайн), чужой skill-комбайн
> `ui-ux-pro-max` (как референс **структуры** брендбука, не как инструмент), и наша система. При
> расхождении приоритет — мировой канон + наша конвенция «ноль дублей». См. «Источники» в конце.

# ЧАСТЬ I — PLAYBOOK (для AI-копии: правила и решения)

## 0. Инструкция ассистенту: как применять этот файл

Когда пользователь хочет дизайн-документ / корпоративный стиль / «чтобы все делали единообразно»:

1. **Собери базу всегда** (§3: бренд-ядро + UI-kit) — её достаточно для одной страницы в стиле.
   **Расширения** добавляй по проекту: ось A (масштаб — много страниц, Storybook) и/или ось B
   (носители — презентация / PDF / визитка). Не раздувай вперёд продукта.
2. **Не «сделай красиво».** Если стиля нет — либо **декомпозируй сильный референс в правила** (§5),
   либо возьми бренд/вкус у пользователя. «Красиво» в лоб = усреднённое «как у всех». **Вкус не
   угадывают** — surface assumption и спроси/попроси референс, если эстетика не задана.
3. **Веди по анатомии** (§2): раздел за разделом. Каждый цвет — с **ролью** и Usage, каждый шрифт — со
   **шкалой**, каждый компонент — с **состояниями**.
4. **Всё на токенах** (§4): роли важнее значений; один источник правды → CSS-переменные/Tailwind.
5. **Проверь по анти-паттернам** (§6) перед сдачей: AI-slop, запреты логотипа, prohibited terms.
6. **Закрепи артефакт в `docs/` проекта** — он не черновик-на-выброс, а постоянный вход для
   производства (`frontend-design`) и сопутствующих документов (презентации, бланки).

Главный приоритет решений: **согласованность через разных исполнителей** (токены + текстовый
контракт) × **минимум, достаточный для проекта** (база сразу, расширения по нужде). Всё остальное — потом.

## 1. Правила-аксиомы (TL;DR всей базы)

1. **Источник правды — токены, не картинка.** ИИ и инструменты работают с именованными правилами
   кратно лучше, чем с макетом в Figma. (§4, §II-B)
2. **Markdown-first, не Figma.** Главный актив для ИИ — текстовая база, к которой подключается любой
   агент. Figma/Storybook — опционально и на масштабе, не на старте. (§II-B)
3. **Не «сделай красиво» → декомпозируй референс в правила.** Возьми сильный образец, вытащи *что
   именно* делает его хорошим, запиши как правила. (§5, §II-C)
4. **Роль цвета важнее значения.** Не «синий», а `primary-500`. Тёмная тема и бренд-варианты — это
   **алиасы** поверх примитивов, а не отдельная палитра. (§4)
5. **База обязательна даже для мелкого проекта.** Бренд-ядро (палитра-роли + типографика + лого +
   токены + voice) + **UI-kit** (базовые компоненты) — дёшево и делает любую отдельную страницу. (§3)
6. **Один документ: база + расширения.** База (ядро + UI-kit) сразу; дальше расширения по осям —
   масштаб продукта и носители (§3), не переписывается с нуля.
7. **Креативный ИИ держит стиль на ~80%.** Его — на черновую визуальную концепцию; **финал** строгого
   стиля собирает дисциплинированный агент **по контракту**. (§II-C)
8. **Бренд-док первичен → токены производны.** Человекочитаемый документ — источник правды; токены
   синкаются из него, не живут как независимый второй файл. (§4)
9. **Исполнитель строит из существующего, не выдумывает.** Дай агенту каталог компонентов/токенов →
   он собирает экран из них, а не плодит дубли. (§8)

## 2. Анатомия дизайн-документа (канонические разделы)

Полный набор разделов. Для конкретного проекта берёшь **столько, сколько требует проект** (база
всегда, расширения по §3) — но порядок и форму держи эту.

| Раздел | Что задаёт | Как фиксировать (форма) |
|---|---|---|
| **Quick Reference** | шпаргалка вверху документа | таблица: primary/secondary цвет · primary шрифт · 3 черты голоса. TL;DR для исполнителя |
| **Логотип** | знак бренда и его защита | варианты (primary / stacked / icon-only / mono) · clear space (= высота знака) · min size (digital ~80px, print ~25mm) · Don'ts |
| **Цветовая палитра** | цвета **по ролям** | 4 группы: **Primary** (бренд/CTA/заголовки) · **Secondary** (акценты) · **Neutral** (фоны/текст/бордеры) · **Semantic** (success/warning/danger/info). Каждый цвет: `Name · Hex · RGB · Usage` |
| **Типографика** | шрифты и шкала | Font Stack (`--font-heading` / `--font-body` / `--font-mono`) · Type Scale таблицей: элемент → шрифт → вес → размер desktop/mobile → line-height. Минимум **3 уровня**: Heading / Body / Secondary |
| **Spacing и сетка** | ритм отступов | база **4/8px** · spacing-scale (xs–2xl) · grid · контейнеры · брейкпоинты |
| **Иконография** | стиль иконок | outline / filled / duotone (одно!) · сетка 24px · толщина штриха · corner radius |
| **Фото / иллюстрации** | визуальный язык изображений | освещение, сюжет, цветообработка фото; иллюстрации — только в палитре бренда, единая толщина штриха |
| **UI-компоненты** | кнопки/поля/таблицы/карточки/модалки/навигация | для каждого: варианты (primary/secondary/danger) + **состояния** (default · hover · active · disabled · error · loading · focus) |
| **Voice & Tone** | как бренд звучит | Personality (3–4 черты) · Voice Chart («We Are / We Are Not») · Tone by Context (marketing/support/error/success) · **Prohibited Terms** с причиной |
| **Motion** | движение | длительности (150–300ms) · easing · допустимые переходы · ограничения (напр. exit ≤ 70% entrance) |
| **Accessibility** | доступность | контраст текста **AA 4.5:1** (крупный/UI — 3:1) · focus-видимость · keyboard · touch-target ≥ 44px |
| **Do / Don't** | правильный vs неправильный пример | сквозь все разделы — короткие пары «так / не так» |
| **Changelog** | версия документа | `v X.Y` · дата · что изменилось |
| **Extractable Fields** *(для ИИ)* | поля для машинного чтения | структура, из которой агент/скрипт берёт `colors.primary`, `typography.heading`, `voice.traits` |

**Разумные дефолты** (подставляй, если у проекта нет своих): type scale H1 `700 48/32px LH1.2` … Body
`400 16px LH1.5` … Caption `12px`; spacing 4/8px; контраст AA 4.5:1 / AAA 7:1; touch 44×44; Primary
Button высота 48px, radius 12px; анимация 150–300ms.

## 3. Что входит в базу, а что — расширения (модель роста)

Документ **не** растёт линейной лесенкой. Есть **база** (создаётся первой, всегда) и **расширения**
по двум независимым осям (добавляются по мере надобности).

**БАЗА — обязательный минимум, которого достаточно, чтобы сделать ЛЮБУЮ ОДНУ страницу в стиле:**

| Слой базы | Что входит |
|---|---|
| **Бренд-ядро** | палитра по **ролям** · типографика (3+ уровня) · логотип-блок · **токены** (primitive→semantic) · voice кратко |
| **UI-kit** | базовые компоненты (кнопки / поля / select / карточки / модалки) со **состояниями** · сетка · spacing · radius · иконки |

С базой уже можно сверстать полноценную **отдельную страницу** и видеть архитектуру стиля. Меньше
базы — нельзя (голое бренд-ядро без UI-kit хватает только для печатных/слайдовых носителей — ось B).

**РАСШИРЕНИЯ — по двум осям, опционально:**

| Ось | Когда нужна | Что добавляется |
|---|---|---|
| **A · масштаб продукта** | одна страница → **много связанных страниц** как единый продукт (портал · CRM · SaaS) | UX-паттерны · правила компоновки многих экранов и навигации · **Storybook** (живой каталог — опция на масштабе: много компонентов / разработчиков) · AI Playbook · governance |
| **B · носители** | та же база на **новый тип артефакта** | надстройка под носитель: презентация (шаблоны слайдов) · PDF / документы · визитка / бланк (печать, форматы) · email · соцсети — все тянут цвет/шрифт/лого из ядра |

> **Ключевое:** портал/CRM/SaaS отличаются от лендинга **не более глубоким дизайном, а количеством и
> связностью страниц** (продукт целиком). Одну страницу из любого SaaS делаешь уже на базе — ради
> этого UI-kit и входит в базу. **Storybook — опция** для каталогизации на масштабе, не часть базы.

> Это **модель роста документа**, НЕ «этапы» процесса 0–7 и НЕ «стадии» кода 0–6 — глоссарий в
> `CLAUDE.md`. Параллель — «фазы роста стека» в `saas-stack-kb.md §8` (минимум сразу, сложное — по
> мере боли).

**Механизм единства носителей (ось B):** визитка, презентация, PDF и сайт берут цвет/шрифт/лого из
**одного ядра** через токены → поэтому и сочетаются. Носитель добавляет лишь тонкую надстройку, не
свой отдельный бренд.

**Два правила старта:**
- **Не с белого листа.** Стартуй с готовой базы (**shadcn/ui** для Next.js, + при желании Magic UI /
  Aceternity) — её хорошо знает ИИ, компоненты кастомизируются, AI-инструменты умеют под неё генерить.
- **Не раздувай вперёд продукта.** База — сразу; ось A (Storybook / паттерны многих экранов) — когда
  продукт реально вырос; ось B — когда понадобился носитель. Storybook на нулевой аудитории = тот же
  overengineering, что K8s на старте.

## 4. Design Tokens (технический фундамент единства)

**Что это.** Именованные переменные дизайн-решений — машиночитаемый слой «истины». Вместо «синий» →
`primary-500 = #2563EB`; вместо «скругление» → `radius-md = 12`.

**Три яруса (канон W3C):**
- **Tier 1 — Primitive** (сырые значения): `color.blue.500`, `space.4`, `font.size.16`.
- **Tier 2 — Semantic** (роли, ссылаются на примитивы): `color.primary`, `color.background`,
  `color.foreground`, `color.border`, `color.surface`.
- **Tier 3 — Component** (привязка к компоненту): `button.primary.background`, `card.padding`.

**Два типа значений:**
- **Alias** — токен ссылается на другой токен. **Тёмная тема, бренд-варианты, темизация = просто
  другие алиасы поверх тех же примитивов** (не инверсия цветов, а другой набор семантических ссылок).
- **Composite** — значение из нескольких под-значений, всегда применяемых вместе (typography = font +
  size + line-height + color).

**Стандарт.** W3C Design Tokens Community Group — первая стабильная версия спецификации вышла в
**окт-2025** (за ней Adobe, Google, Microsoft, Meta, Figma, Salesforce, Shopify). Один файл токенов
генерирует код под web/iOS/Android — единый источник для дизайнера, разработчика и ИИ.

**Выход в наш стек:** токены → **CSS-переменные** → **Tailwind v4** (`@theme`) + **shadcn/ui**.
Правило: в коде только `var(--token)`, **никаких хардкод-цветов/шрифтов**.

**Принцип «бренд-док → токены»:** человекочитаемый документ — первичен; токены синкаются **из него**
(вручную или скриптом), а не существуют как независимый второй источник, который рассинхронизируется.

```jsonc
// мини-пример 3-слойных токенов (reference-синтаксис + dark как override semantic-слоя)
{
  "primitive": { "color": { "blue":  { "500": "#2563EB" },
                            "gray":  { "50": "#F9FAFB", "900": "#111827" } } },
  "semantic":  { "color": { "primary":    "{primitive.color.blue.500}",
                            "background":  "{primitive.color.gray.50}",
                            "foreground":  "{primitive.color.gray.900}" } },
  "component": { "button": { "primary": { "background": "{semantic.color.primary}" } } },
  "dark":      { "semantic": { "color": { "background": "{primitive.color.gray.900}",
                                          "foreground": "{primitive.color.gray.50}" } } }
}
```

## 5. Процесс создания дизайн-документа (шаги)

**Шаг 0 — Вход.** Возьми контекст продукта: PRD (`docs/product.md`), какие экраны/носители нужны,
аудитория, желаемый тон. Без контекста результат — усреднённый.

**Шаг 1 — Стиль.** Либо у пользователя есть бренд → бери его. Либо стиля нет → **декомпозируй
референс**: возьми сильный образец (Dribbble / Behance / сайт-конкурент), ИИ парсит его и вытаскивает
*что именно* делает его привлекательным → правила + черновой `style-guide.md`. **Не** «сделай красиво».

**Шаг 2 — Палитра по ролям.** Назначь Primary + Neutral + Semantic (+ Secondary при нужде). Роли, не
просто цвета. Сразу проверь контраст пар текст/фон.

**Шаг 3 — Типографика.** Heading / Body / (Mono) + шкала. Inter допустим, но **правь letter-spacing**
(дефолтный Inter — признак AI-slop).

**Шаг 4 — Лого-блок.** Варианты + clear space + min size + Don'ts — даже если логотип простой.

**Шаг 5 — Токены.** Оформи решения шагов 2–4 как `primitive → semantic` (§4).

**Шаг 6 — Компоненты + состояния.** Собирай/проверяй на **насыщенном** экране (dashboard, рабочий
экран) — чтобы поместилось много компонентов и можно было отщепить компонентную базу с состояниями.

**Шаг 7 — Чек-лист.** Прогон по анти-паттернам (§6): AI-slop, контраст, hug-vs-fixed, запреты лого,
prohibited terms.

**Шаг 8 — Закрепи.** Положи артефакт в `docs/` проекта. Опционально на масштабе (ось A) — зеркало в
Figma и/или Storybook (синхронизируются с теми же токенами).

**Дисциплина процесса:** итеративно, с критикой (несколько прогонов, а не «за 5 секунд»); креативный
ИИ-генератор — только на черновую концепцию; **финал собирает дисциплинированный агент по контракту**;
держи рабочую среду структурированной (отдельно контекст, конкуренты, DS), чтобы агент шёл точечно в
нужный файл и не жёг контекст.

> **Легаси без системы** (напр. 200 экранов, собранных вразнобой): сначала бегло проставь нейминг
> экранов → раздроби на user-flow/группы → иди группами по 1–2 → сначала **описательная** DS
> (правила/паттерны), потом визуализация. На хаотичном материале агент «сыпется» — часть причёсывай
> руками.

## 6. Анти-паттерны (что НЕ делать)

### AI-slop — по чему палится ИИ-дизайн (вычищать!)
- ❌ **Бордеры везде.** ИИ «обожает обводки». → давай объём/структуру, а не рамки.
- ❌ **Дефолтный Inter без правки letter-spacing.** Живой дизайнер почти всегда правит интервал.
- ❌ **Теги/пилюли на каждом шагу** · онлайн-индикатор с ореолом/свечением (подложка того же цвета).
- ❌ **Liquid glass / стекло** — красиво в референсе, но нормально не сверстать (много наложений,
  тяжёлые библиотеки); дорого по времени, обычно выходит «приземлённо».
- ❌ **Generic-композиция:** стоковый hero (центр + gradient blob + дефолтный CTA) · card-grid без
  иерархии · немодифицированные дефолты библиотек · одинаковые radius/shadow/spacing везде.

### Запреты логотипа
растягивать · сжимать · вращать · скашивать · тени/градиенты/прозрачность/обводки · менять цвета вне
палитры · кадрировать или переставлять части · класть на «занятый» фон без проверки контраста ·
неверный формат под носитель (веб = SVG, печать = PDF/вектор).

### Запреты цвета / тем
контраст текста < 4.5:1 в любой теме · бордеры/разделители, видимые только в одной теме · слабый scrim
модалки (нужно 40–60% непрозрачности) · **инверсия** цветов для dark вместо десатурированных вариантов
· передавать смысл **только цветом** (нужен ещё значок/текст).

### Запреты voice (prohibited terms)
`Revolutionary` · `Best-in-class` · `Seamless` · `Synergy` · `Leverage` (→ «use») — overused / vague /
jargon. У каждого проекта — свой список, но эти попадают почти всегда.

### Правило дисциплины
**Дизайн-систему нельзя «интерпретировать иначе».** Кнопка, радиус, оттенок — одинаковы **всегда**. В
этом весь смысл консистентности (и именно здесь креативный ИИ сдаёт со своими 80% — потому финал по
контракту).

## 7. Инструменты по шагам

| Шаг / задача | Инструмент | Статус |
|---|---|---|
| Документ-контракт (правила, do/don't) | **Markdown** | **обязательно — ядро** |

| Декомпозиция референса в правила | **ИИ (Claude)** парсит картинки референса | по нужде |

**Наши скиллы — где какой (не дублируем, ссылаемся):**
- `document-skills:canvas-design` — создать **визуальный артефакт** (постер/лого-концепт) + design
  philosophy. На шаге генерации концепции/ассетов.
- `document-skills:theme-factory` — быстро применить готовую тему к **сопутствующему документу**
  (презентация/док), когда нет своего полного стиля.
- `document-skills:brand-guidelines` — применить **Anthropic**-стиль к артефакту (если нужен именно
  он). Для **своего** бренда — наш дизайн-документ, не этот скилл.
- `document-skills:frontend-design` — **реализация** дизайн-документа в код (стадия 2 кода, см.
  `rules/frontend.md`). **Не** на этом этапе — это потребитель нашего артефакта, а не его создатель.

## 8. Механизм единства — как один документ держит разных исполнителей

1. **Токены = технический механизм.** Каждый исполнитель (человек и ИИ) берёт цвет/радиус/отступ из
   **токена**, а не подбирает. Меняешь токен в одном месте → меняется у всех. Это и есть «гармонично
   сочетается».
2. **Markdown-контракт «делать / НЕ делать»** одинаково читают человек и ИИ-агент (в т.ч. через MCP).
3. **Каталог компонентов** (текстовая карта / Storybook): исполнитель **строит из существующего**, не
   выдумывает. Промпт не «создай CRM-экран», а «используй DataTable, LeadCard, FilterBar».
4. **Библиотека промптов** — готовые шаблоны под задачи, ссылающиеся на документ. Пример:
   *«Используй наш Design Core. Создай экран управления сделками. Компоненты: DataTable v2,
   FilterBar v1, Sidebar v1»*.
5. **AI Playbook** — короткий текст, который агент получает **первым** перед проектированием. Пример:
   *«Все интерфейсы: светлые, минималистичные, максимум 1 акцент, таблицы приоритетнее карточек,
   shadcn, не более 2 теней»*.
6. **Креативный ИИ ~80% → финал по контракту.** Концепцию можно генерить свободно; строгий стиль на
   проде собирает дисциплинированный агент по документу-контракту.

> **Шаблон-рыбу артефакта** «Brand & Design Core» (заполняемый каркас по §2: база + расширения §3)
> см. в `dev-workflow-kb.md` (шаблоны проекта, рядом с PRD/ADR) — справочник остаётся «знанием»,
> шаблон живёт там, где остальные шаблоны (ноль дублей).

# ЧАСТЬ II — ОБЪЯСНЕНИЯ (для человека: «что это и почему»)

## A. Семейство документов: brand book vs style guide vs design system vs tokens vs playbook

Это **не один документ**, а слои, различающиеся вопросом и аудиторией:

| Документ | Отвечает на вопрос | Для кого | Покрывает |
|---|---|---|---|
| **Brand Book / Guidelines** | как выглядит и звучит бренд | маркетинг, подрядчики | логотип, цвета, шрифты, иконки, фото, tone of voice + носители (визитки, баннеры, презентации) |
| **Style Guide** | какой визуальный стиль у интерфейса | дизайнеры, контент | концентрированный визуальный срез: примеры экранов, карточек, таблиц. Узкий, «как выглядит» |
| **Design System** | как строятся интерфейсы | дизайн + разработка | самый широкий: токены + библиотека компонентов + правила взаимодействия + доступность |
| **Design Tokens** | чему равны значения стиля | код, инструменты, ИИ | машиночитаемый слой правды (цвет/отступ/радиус как переменные) — фундамент под всё |
| **AI Design Playbook** | как должен выглядеть ЛЮБОЙ продукт компании (+ инструкция для ИИ) | все исполнители, включая Claude/Cursor | философия + принципы + дизайн-система + примеры + готовые промпты |

Наш **«Brand & Design Core»** — один документ: **база** (бренд-ядро + UI-kit) + **расширения** по двум
осям (масштаб продукта и носители, §3). Не нужно выбирать «или-или» между типами выше — собираешь базу
и добавляешь только нужное.

## B. Зачем токены и почему markdown, а не Figma

ИИ-агенты и инструменты сборки работают с **правилами и текстом** кратно лучше, чем с картинками из
Figma. Картинку нужно «разглядеть» и интерпретировать (и каждый интерпретирует по-своему — отсюда
рассинхрон). Токен `primary-500 = #2563EB` интерпретировать нельзя — он однозначен. Поэтому главный
актив компании для эпохи ИИ — **текстовая база** (Markdown + токены в Git/Notion/Obsidian), к которой
подключается любой агент, а **не** файл Figma. Figma остаётся полезной как зеркало/макет для людей на
больших продуктах (ось A — много экранов / команд), но **источник правды** — текст и токены.

## C. Почему «декомпозиция референса», а не «сделай красиво»

Если попросить ИИ «сделай красивый дизайн» в лоб — он выдаёт **усреднённое** по всему интернету «как у
всех». Сильный фирменный стиль получается иначе: берёшь конкретный сильный образец, **раскладываешь
его на правила** (какая сетка, какой контраст, какой ритм, что делает его дорогим), и работаешь
**итеративно** с критикой. Креативный ИИ-генератор полезен для черновой концепции, но строгую
систему он держит лишь на ~80% — поэтому финал собирает дисциплинированный агент по зафиксированным
правилам, а не «креативщик» на лету.

## D. База и расширения простыми словами

Сначала собери **базу**: бренд-ядро (цвет / шрифт / лого / токены / voice) + UI-kit (базовые
компоненты, сетка, радиусы, иконки). База почти бесплатна, даёт узнаваемость и **с неё уже верстаешь
любую отдельную страницу**. Дальше добавляй только нужное: продукт вырос в много связанных страниц
(портал / CRM / SaaS) → **ось A** (паттерны, Storybook-каталог); понадобился новый носитель
(презентация / PDF / визитка) → **ось B** (надстройка поверх той же базы). Storybook и полную систему
на нулевой аудитории не строй — это overengineering.

## E. Как стыкуется с нашим стеком и `frontend-design`

Наш дефолтный стек (`saas-stack-kb.md`) — Tailwind v4 + shadcn/ui — ложится на токены идеально: токены
становятся CSS-переменными в `@theme`, а компоненты shadcn копируются в проект и кастомизируются под
них. Связка этапов: **на идея-план рождается дизайн-документ** (этот справочник) → **на стадии кода
`frontend-design` его реализует** в распознаваемый, не-generic UI. Так `frontend-design` (который сам
по себе работает только в производстве) наконец «участвует в плане» — опосредованно, через документ,
который он потом исполняет. Производственные проверки (доступность, перф, web-guidelines) остаются за
`a11y-audit` / `web-performance` / `web-design-guidelines` — справочник их не дублирует.

## Источники

1. **Мировые практики дизайн-систем** (research-диалог + W3C Design Tokens, Material/Atlassian/Carbon,
   Atomic Design) — типы документов, канон разделов, 3-ярусные токены, уровни зрелости, механизм
   единства через токены + markdown + Storybook. *(приоритетный — канон)*
2. **Обучающий урок** (декомпозиция стиля + AI-пайплайн Claude→Figma→Storybook) — принцип «не сделай
   красиво, а декомпозируй референс», markdown-контракт, чек-лист AI-slop, креативный ИИ = 80%.
   *(приоритетный — практика)*
3. **`ui-ux-pro-max-skill`** (чужой комбайн) — как **референс структуры** брендбука (Quick Reference →
   Color → Type → Logo → Voice → Imagery → Components → Changelog), принцип «бренд-док → токены»,
   списки «нельзя». Взяты структура и принципы; CSS-магазины палитр / код-ревьюер / генерация ассетов
   / мульти-платформенный CLI — НЕ взяты (избыточный комбайн, дубль наших скиллов). *(референс)*
4. **Наша система** — `saas-stack-kb.md` (формат справочника, стек), `rules/frontend.md`
   (`frontend-design` на производстве), `dev-workflow-kb.md` (маршрут этапов). Конвенция «ноль
   дублей»: производство не дублируем, ссылаемся. *(контекст интеграции)*

При расхождении: мировой канон (1) + наша конвенция «ноль дублей» (4) имеют приоритет. Зафиксированные
решения: артефакт = **один растущий документ** (не семейство файлов); охват справочника = **узкий**
(создание документа на идея-план; производство — за существующими скиллами).
