# Dev Workflow KB — стандарт PRD и документации проекта

> **Что это.** Единый рабочий процесс для **любого нового проекта**: как идти от идеи к коду и какие
> документы вести.
>
> **Зачем.** Чтобы не было «бардака» (разные форматы, устаревшие доки, хватание то за один инструмент,
> то за другой). Один хребет, одна конвенция, ноль дублей.
>
> **Серия KB:** этот файл — про *процесс*. Соседние в `spravochniki/`: `idea-kb.md` (этап 0),
> `saas-stack-kb.md` (выбор стека), `dev-coding-process-kb.md` (как кодим). Маршрут зовёт нужный на этапе.
>
> Версия 1.0 — 2026-06-05.

---

## 0. Как этим пользоваться

**Тебе (владельцу):** на новом проекте скажи копии Claude одно из:
- «ведём проект по этому потоку» — пройдём все этапы по порядку;
- или назови этап: «сделаем PRD по стандарту», «нужен ADR на это решение», «оформи docs/».

**Копии Claude (ассистенту):** этот файл — твой процесс. Держи поток §2, конвенцию §3, бери шаблоны §4.
На каждом этапе бери инструмент(ы) **из таблицы** — не импровизируй с альтернативами извне (правило
против «хватания», §1). На большинстве этапов он один; на ревью (§5) и доках (этап 6) их несколько.
Документируй решения (§4 ADR), а не пересказ кода.

---

## 1. Принципы (хребет)

1. **Один process-хребет — `superpowers`.** Не подключать второй фреймворк (это и был источник «хватания»).
2. **Context precedes code** — что и зачем определяем ДО реализации.
3. **Документируй решения и «почему», не код.** Код показывает «как»; «почему выбрали X, отвергли Y» —
   только в доках (ADR).
4. **Живое обновляй, снимки не трогай.** `product.md`/`CLAUDE.md` — живые; ADR и session-логи — снимки.
5. **Не усложнять (karpathy).** Минимум, что решает задачу. Никаких спекулятивных абстракций/слоёв.

## 2. Поток проекта (этап → инструмент → артефакт)

```
ИМПУЛЬС ─▶ ИДЕЯ ─▶ PRD ─▶ ДИЗАЙН ─▶ ПЛАН ─▶ КОД ─▶ ДОКИ ─▶ ФИНАЛ
```

| # | Этап | Инструмент (из таблицы) | Артефакт на выходе |
|---|------|-------------------|--------------------|
| 0 | **Формирование идеи** — из сырого импульса в идею (нужна ли она) | `idea-kb.md` | `idea-doc` + вердикт go/no-go |
| 1 | **Брейншторм + черновой дизайн** — раскрытие идеи + техника | `superpowers:brainstorming` (до его spec-gate; в `writing-plans` НЕ уходить) | черновой design doc (промежуточный, удаляется на этапе 3) |
| 2 | **PRD** — продуктовый срез (техника НЕ протекает) | `document-skills:doc-coauthoring` + шаблон §4.1 | `docs/product.md` |
| 3 | **Дизайн + ADR** — формализация техники brainstorming (2-й проход: challenge + alternatives) | шаблон §4.2 (ADR) + §4.3; стек — из `saas-stack-kb.md` | `docs/tech-stack.md`, `docs/decisions/ADR-*.md`; промежуточный design doc удалить |
| 4 | **План** — шаги реализации (большая фича: ТОЛЬКО после этапов 2-3) | `superpowers:writing-plans` | план задач |
| 5 | **Код** (разворот → стадии 0–6, `dev-coding-process-kb.md`) | `superpowers:executing-plans` + TDD + `verification-before-completion` | рабочий код + тесты |
| 6 | **Доки по ходу** | ADR на решения · `knowledge.md` на грабли · `claude-md-management:revise-claude-md` для CLAUDE.md | обновлённые доки |
| 7 | **Финал** | `/code-review` (см. §5) + `superpowers:finishing-a-development-branch` | смерженная работа |

**Память сквозь всё** — встроенная auto-memory (`~/.claude/projects/<проект>/memory/`). Факты копятся сами.

**Брейншторм → середина → план (важно).** `superpowers:brainstorming` доходит до своей спеки и по своему
gate ждёт одобрения, затем хочет в `writing-plans`. У нас его design doc — **ЧЕРНОВИК**. Для **большой**
фичи: после brainstorming обязательны этапы 2-3 (PRD + дизайн/ADR по нашему стандарту, `docs/`),
проходящие ПОВЕРХ его техники (2-й проход усиливает); `writing-plans` (этап 4) — только после них;
**одобрение спеки brainstorming ≠ «писать план»**; промежуточный design doc удаляется после переноса в
`docs/`. Для **малой** фичи: середина пропускается — brainstorming уходит в `writing-plans` как есть.

Этап 0 — только для сырой/новой идеи (нужна ли она вообще); на уже ясной идее начинай с 1.
Что считать «ясной» — **входной тест в `idea-kb.md` §1**: можешь без сомнений назвать юзера + спрос +
wedge → ясная (начинай с 1); иначе → этап 0.
Не каждый проект проходит все 7. **Масштаб — спроси явно:** в начале агент ЯВНО задаёт вопрос «это
большая или малая фича?» и по ответу выбирает путь (сам НЕ классифицирует). Спросить невозможно
(автономный запуск: cron/headless/исполнение плана без человека) → классифицируй по критериям сам и
пометь выбор как допущение в плане/отчёте. **Большая** → полный поток с
серединой: brainstorming → PRD → дизайн+ADR → план (этапы 1-2-3-4). **Малая** → облегчённый: brainstorming
→ план (1→4), середина (PRD/дизайн) пропускается. Демо/POC: облегчённый PRD. Продукт под клиента:
полный поток + чистый PRD без кишок архитектуры (его читает заказчик).

## 3. Конвенция документации (одна на все проекты)

```
docs/
├── idea.md          ← idea-doc этапа 0 (опционально — если идея оформлялась через idea-kb)
├── product.md       ← PRD: что/зачем/персоны/success-criteria/scope + non-scope
├── tech-stack.md    ← стек и архитектура (ссылка на saas-stack-kb.md, не дублировать)
├── decisions/       ← ADR-NNNN-<slug>.md — почему так (с alternatives)
└── knowledge.md     ← паттерны проекта + грабли (institutional memory)
CLAUDE.md            ← правила поведения для Claude (живой документ)
auto-memory          ← факты между сессиями (автоматически, не вручную)
```

Дисциплина против устаревания (облегчённая): `product.md`/`tech-stack.md` —
источник правды, обновляются ПЕРВЫМИ при смене решения; остальное приводится в соответствие; не ссылаться
на устаревшую цену/модель/решение. Старые ADR не удалять — помечать `Superseded by ADR-NNNN`.

> **`docs/superpowers/`** — рабочие артефакты superpowers: `specs/` (черновой design doc brainstorming)
> удаляется после переноса в `product.md`/`tech-stack.md`/`decisions/` и **не источник правды**;
> `plans/` (план writing-plans) — рабочий, живёт пока исполняется.

## 4. Шаблоны (копируй и заполняй)

### 4.1 PRD (`docs/product.md`)
```markdown
# PRD — <проект>
> Статус: draft / на ревью / принят. Дата. На основе: <ТЗ/брейншторм>.

## 1. Обзор
- Что это (1-2 фразы) · Зачем (проблема) · Критерий успеха · Заказчик/для кого
## 2. Пользователи и роли
- <роль>: точка входа, что делает, что видит / НЕ видит
## 3. Функции
- 3.1 <блок> — что делает, как работает (без «как» на уровне кода)
## 4. Нефункциональные требования
- нагрузка · приватность · латентность · бюджет
## 5. Границы
- Что НЕ входит (non-scope — половина ценности PRD здесь)
## 6. Риски и открытые вопросы
- <риск> → реакция
```

### 4.2 ADR (`docs/decisions/ADR-NNNN-<slug>.md`) — из Cortex-образца
```markdown
# ADR-NNNN: <заголовок>
## Status: Proposed / Accepted / Superseded by ADR-NNNN
## Context: какую проблему решаем, какие ограничения/силы
## Options considered:
| Опция | Плюсы | Минусы |
|-------|-------|--------|
| A     |       |        |
| B (do nothing) |  |       |   ← всегда включай «ничего не делать»
## Decision: выбранная опция (императивно)
## Rationale: почему именно она — со ссылкой на ЭТОТ контекст, не на «best practice»
## Consequences: что станет проще / сложнее; риски
## Revisit when: при каких условиях это решение станет неверным
```
> Правило (Cortex): *«ADR без раздела Options — это пресс-релиз, а не решение».* Перечисляй ВСЕ опции,
> что рассматривал, не только выбранную. Документируй trade-offs честно — ни одна опция не «вся в плюсах».

### 4.3 System-design (для нетривиальной архитектуры — `docs/decisions/` или раздел в product)
```markdown
## Дизайн: <система>
### Scope: что именно проектируем (только correctness-critical ядро)
### Компоненты: таблица (компонент → назначение)
### Схема: ASCII/mermaid — потоки данных, границы
### Ключевые решения: ссылки на ADR
### Failure modes: что ломается и как обрабатываем
```

### 4.4 Knowledge (`docs/knowledge.md`)
```markdown
## Паттерны (что применяли — переиспользовать)
- <паттерн> — суть, где использован
## Грабли (что ловили — не повторять)
- <ошибка> — симптом, причина, как избежать
```

## 5. Code-review — НЕ один инструмент, у каждого своя роль

Они НЕ дублируют друг друга. Бери по задаче; для серьёзной — прогоняй по очереди; безопасность не пропускай.

| Инструмент | Когда |
|---|---|
| `/code-review` | основной — баги + чистка диффа. `ultra` = глубокий облачный multi-agent для важного |
| `/simplify` | только упрощение/качество, без поиска багов — быстрая косметика |
| `/security-review` | **безопасность — всегда для серьёзных задач, не пропускать** |
| `superpowers:requesting-code-review` | как корректно запросить ревью (процесс) |
| `superpowers:receiving-code-review` | как реагировать на фидбэк — проверять, не соглашаться слепо |

**Правило:** мелочь → `/code-review`. Серьёзная фича/перед мержем → `/code-review` + `/security-review`
(+ `/simplify` для чистоты). Чувствительное (auth/платежи/данные) → `/security-review` обязательно.

## 6. Система знаний: маршрут + справочники (КАК расширять)

Эти KB-файлы устроены по одному принципу — чтобы не плодить структуры и не дублировать.

- **Маршрут** (этот файл) — тонкий: этапы, порядок, ссылки. Всегда «с собой».
- **Справочники по ТЕМАМ** — каждый про одно; маршрут зовёт нужный на нужном этапе. Грузишь только
  актуальный — контекст не раздувается.

Аналогия: маршрут = рецепт (шаги); справочники = техники («как нарезать»). Рецепт *ссылается* на технику,
а не переписывает её внутри себя.

**Правило расширения (фиксируем, чтобы не изобретать структуру заново):**
1. Всплыла новая тема (разработка, тесты, безопасность, деплой, поддержка…) → **дополни существующий
   справочник по теме или заведи новый**. НЕ создавай файл «по этапу», НЕ копируй сквозное в несколько мест.
2. **Сквозная тема = ОДИН справочник.** Безопасность нужна на дизайне И на ревью И в поддержке — но живёт
   в одном `security-kb`, маршрут ссылается на него с разных этапов. Ноль дублей.
3. **Справочник рождается, когда дошёл до темы**, не все сразу (иначе over-engineering).
4. Делим **по темам, НЕ по этапам** (этапы порождают дубли сквозного — это и есть ловушка).

**Реестр справочников:**

| Тема | Файл | Статус |
|---|---|---|
| Формирование идеи (этап 0) | `idea-kb.md` | ✅ есть |
| Выбор стека / build-vs-buy | `saas-stack-kb.md` | ✅ есть |
| Образцы оформления ADR/spec | Cortex `.claude/skills/architecture/` (github.com/cdeust/Cortex, MIT) | 📎 внешний |
| Процесс разработки (как кодим) | `dev-coding-process-kb.md` + `~/.claude/rules/<направление>.md` | ✅ есть |
| Тестирование · Безопасность · Поддержка | — | ⏳ по мере захода в тему |

> При добавлении нового справочника — впиши его в этот реестр и сошлись на него из нужного этапа §2.

## 7. Анти-паттерны (чего не делать)

- ❌ Подключать второй process-фреймворк рядом с superpowers (вернётся «хватание»).
- ❌ Документировать пересказ кода вместо решений и «почему».
- ❌ ADR без alternatives. PRD без non-scope. Дизайн без failure-modes.
- ❌ Держать устаревший документ как актуальный (обновляй живое при смене решения).
- ❌ Тяжёлая оркестрация (board/loops/tracks), спекулятивные слои — over-engineering для solo.
- ❌ Дублировать `saas-stack-kb` в каждый проект — ссылаться.

## Итог в одну строку

**Новый проект: brainstorming → PRD (`product.md`) → дизайн (ADR) → план → код+тесты → доки по ходу →
ревью. Один хребет superpowers, доки по конвенции `docs/`, стек из saas-stack-kb. Документируй «почему»,
не усложняй.**
