# Процесс разработки (как кодим)

Стадии работы над кодом — от ориентации до отгрузки. Один цикл и для нового кода, и для правок готового продукта.
**Эти «стадии» 0–6 — разворот этапа «Код» (этап 5) процесса `dev-workflow-kb.md`.** Термины: «стадия» = только здесь (код 0–6); «этап» = процесс 0–7; «фаза роста» = масштаб стека (`saas-stack-kb.md` §8). Глоссарий — `CLAUDE.md`.

```
0 Сориентироваться ─▶ 1 Оформить ─▶ 2 Написать(TDD) ⇄ 3 Проверить ⇄ 4 Отладить ─▶ 5 Ревью ─▶ 6 Завершить
└────────── сквозь все: A Дисциплина · B Стандарт «готово» · C Безопасность · D Доки/знания ──────────┘
```

Ядро — петля **2 ⇄ 3 ⇄ 4**. Вход на стадии 0 ветвится (новый код / правка готового), дальше путь общий.

## Стадии

Формат: **что делаем · готово, когда · нельзя.**

### 0. Сориентироваться
**Что делаем.**
- **Стандарт проекта — ПЕРВЫМ (и для новой фичи, и для ремонта).** Прочитать `docs/`, если есть: `product.md` (что/зачем), `tech-stack.md` (стек — обязателен к соблюдению), `decisions/` (принятые ADR — не нарушать без нового ADR `Superseded by`), `knowledge.md` (грабли). Работа идёт ПО этому стандарту; нет `docs/` → стандарт ещё не спроектирован, вернись на этап `1-plan/`.
- Новый код: подтвердить, что план, дизайн и контракты на руках.
- Правка готового: прочитать существующее — нужную подсистему, связи модулей, git-log; где ляжет правка, кто её потребители.
- Назвать ставки (**High / Medium / Low**) по объективному признаку — критерии в `coding-discipline.md` (Move 7; вкладывается диспетчером на стадии план/код).
- Сформулировать критерий «готово» одной проверяемой фразой; назвать допущения.

**Готово, когда.** Ставки названы по признаку · критерий «готово» можно проверить · для чужого кода есть объяснение «что трогаю» своими словами.

**Нельзя.** Писать, не назвав слой и ставки · менять код, который не можешь объяснить.

### 1. Оформить изменение
**Что делаем.**
- Назвать слой (core/domain/infra/handlers), куда ложится код. Зависимости — внутрь; наружу не импортируем.
- Записать контракт прежде тела: сигнатура и типы, pre/post-условия одной фразой, инвариант для сайд-эффектов. Для интерфейса — проверка подстановкой: pre можно ослабить, post можно усилить, иначе нарушение контракта.
- Большой объём — разложить: на каждый шов веская причина, граф зависимостей внутрь без циклов, путь миграции по шагам.
- Concurrency (async/locks/shared-state) или формально-критичное (crypto/числа/протокол) — пометить «нужен специалист» до кода.

**Готово, когда.** Слой назван одним словом · у каждой нагруженной функции есть контракт · все зависимости направлены внутрь.

**Нельзя.** Тело раньше сигнатуры и контракта · нетипизированные dict/map через границу слоёв.

### 2. Написать через тест (TDD)
**Что делаем.**
- Тест из контракта первым (red) → минимальная реализация (green) → рефактор для ясности. Тест обязан падать без реализации.
- 7 запретов по умолчанию (обход — только с обоснованием в месте использования) — список в `coding-discipline.md` (Move 3).
- Одна ответственность на функцию (I/O vs вычисление, policy vs mechanism). Корректность сложной функции = произведение корректностей → разделить.
- Простейшее, что решает задачу; только то, что просили.

**Готово, когда.** Тело следует из контракта · тест падает без кода и проходит с ним · проще уже некуда · лишнего нет.

**Нельзя.** Тавтологичные тесты (зелёные без реализации) · core импортирует infra · функция-свалка `utils`.

### 3. Проверить поведением
**Что делаем.**
- Прогнать тесты и **запустить, посмотреть** реальное поведение против критерия «готово». Статика: типы, линт.
- Покрытие по критичности. Явно назвать, **что тесты не доказывают** (concurrency-интерливинги, числовые и adversarial-края) — для этого методы сильнее тестов.
- Факт, на котором стоит код (константа, порог, «все знают»): 0–1 источник — гипотеза, 2+ независимых первичных — провизорно. Нет источника → «не знаю», стоп.
- Красный → стадия 4 или назад в 2.

**Готово, когда.** Каждое утверждение о коде стоит на тесте, замере, типе или ассерте. «Думаю, работает» — гипотеза, не готовность.

**Нельзя.** Зелёные тесты как единственный аргумент для кода, чьи отказы они не покрывают · константа без источника.

### 4. Отладить системно
**Что делаем.**
- Репро (нет репро — нет фикса) → инструментировать до гипотезы (лог/брейк/ассерт) → измерить по всем правдоподобным компонентам, не только подозреваемому → назвать аномалию → бисект до носителя → подтвердить вторым независимым методом.
- Назвать причину, ровно одну: (a) неверный контракт · (b) нарушение слоя · (c) спутанные ответственности · (d) конструкт убил локальное рассуждение · (e) дисциплина ниже ставок. Чинить в источнике.
- Один баг — один фикс — одно изменение.

**Готово, когда.** Причину подтвердили инструментом и вторым методом · репро после фикса проходит · регрессий нет.

**Нельзя.** Заплатка (guard/null-check/try-catch на месте throw) без анализа причины · чинить соседнее в том же фиксе.

### 5. Ревью
**Что делаем.**
- Само-ревью диффа, затем ревью: аудит слоёв (импорты в верную сторону), SOLID, анти-паттерны, безопасность. Каждое замечание — принцип + file:line + важность, без шума.
- Чувствительное (auth/деньги/данные): threat-model, OWASP, supply-chain, defense-in-depth; каждое — CWE и, по возможности, репро.
- Фидбэк проверять, не соглашаться слепо.

**Готово, когда.** Каждое замечание называет принцип · ревью по реальному диффу, не по описанию · критичное разобрано раньше косметики.

**Нельзя.** «LGTM» без чтения · стиль-нитпики топят важное · замечание без принципа.

### 6. Завершить
**Что делаем.**
- Финальная самопроверка: правила соблюдены · контракты на месте · слой не нарушен · локальное рассуждение цело · тесты зелёные · назвать топ-3, что ещё может опровергнуть изменение.
- Вся сюита зелёная и критерий «готово» доказан → мерж/PR.

**Готово, когда.** Все проверки прошли или есть явная передача специалисту · критерий «готово» доказан замером/тестом.

**Нельзя.** Отгрузка с проваленной проверкой без передачи · мёртвый код и «на будущее» без вызова.

## Сквозь все стадии

- **A. Дисциплина.** Слой · контракт · 7 запретов · причина-в-корне · разделение ответственностей · самопроверка · ставки + отказы по умолчанию. Полные определения и список отказов — `coding-discipline.md` (Moves 1–7; вкладывается диспетчером на стадии план/код; единственный источник дисциплины).
- **B. Стандарт «готово».** Логично · проверяемо · по ставкам · только нужное · долг доказательств. Развёрнуто — `coding-discipline.md` (раздел B).
- **C. Безопасность.** Не стадия, а линза: вход — поверхность угроз; письмо — запреты и валидация на границах; ревью — OWASP/CWE.
- **D. Доки/знания.** ADR на решения (с alternatives), knowledge на грабли — по ходу, не в конце. Пишем, почему выбрал X и отверг Y, а не пересказ кода.

## Привязка инструментов по направлениям

Спина выше — общая, без привязки к стеку. Инструменты по направлению (React, Python, бэк, БД, AI) живут в отдельном файле `~/.claude/rules/<направление>.md` (глобально): для каждой стадии — какой скилл или агент применять. Файлы привязок **вкладываются диспетчером** (хук `UserPromptSubmit`) по ключевым словам и расширениям файлов в чате — не грузятся все сразу; объявление направления выбирает, по какой таблице действовать.

## Координация полиглот-фичи (кто ведущий)

Когда фича задействует **несколько направлений** (типичный SaaS = фронт + TS/Node + БД), нужен один ведущий — иначе куски не стыкуются. Куски разваливаются не на «агент плохо написал свой слой», а на **стыках и отсутствии сквозной проверки**.

**Ведущий — это главная сессия** (та, что говорит с пользователем), в роли tech-lead. Роль-агенты — **листья-специалисты**: вызываются через Agent tool, у каждого свой контекст; субагент не оркеструет других субагентов (вложенность — один уровень), поэтому дирижирует только главная сессия.

**Протокол (при >1 направлении):**
1. **Контракты на стыках — ПЕРВЫМИ.** До делегирования ведущий фиксирует контракт каждого стыка (тип API фронт↔бэк, схема новой колонки БД, формат события). Специалист получает не «сделай профиль», а «вот контракт — реализуй свою сторону». ← главный рычаг качества полиглота.
2. **Делегируй кусок профильному роль-агенту** с этим контрактом и границей (`rules/<направление>.md`).
3. **Ведущий интегрирует и сам гоняет сквозной критерий** — три зелёных гейта направлений ≠ работающая фича; «у меня зелено» от специалиста не заменяет интеграцию.
4. **«Готово» фичи** = гейты каждого направления (`rules/<dir>.md`) **и** сквозной критерий стадии 6, проверенный ведущим.
5. **Два человеко-чекпоинта:** одобрить план + контракты (чтобы не построили не то) → автономное исполнение → приёмка результата.

**Кто что решает:** продуктовое (что / приоритет / «достаточно ли хорошо») → пользователь · архитектурное (слой / контракт / стек / граница) → ведущий, фиксирует ADR с alternatives, стратегическое эскалирует пользователю · тактическое внутри направления (как именно написать) → роль-агент в рамках контракта.

Для **одной фичи за раз** этого достаточно — без board/tracks/swarm (тяжёлая оркестрация = over-engineering для solo, см. анти-паттерны `dev-workflow-kb.md`).

**Эскалация в «specialist (TBD)».** Роль-агенты на краю компетенции (формально-критичное, доказательства над concurrency-интерливингами, capacity-планирование) эскалируют в «specialist (TBD)» — **такого субагента в системе нет**. Для ведущего это сигнал, а не тупик: вынеси пробел пользователю явно, возьми внешний second-opinion (`!codex`), или иди best-effort с явным флагом риска в отчёте. Не делай вид, что специалист существует и «всё под контролем».
