# Try before asking the user to act — do it yourself if you can

Сначала попробуй выполнить сам. Только если не получилось — попроси пользователя.

## Триггер

Я готов написать пользователю одну из фраз:
- «запусти X»
- «открой Y»
- «скопируй Z в файл N»
- «выполни команду W»
- «проверь результат и сообщи»
- «установи зависимость»

И при этом действие **физически доступно мне** через имеющиеся инструменты (Bash, Edit, Write, Read, mcp tools).

## Действие

1. **Сначала — выполнить самому.** Через Bash / Edit / Write / соответствующий MCP tool.
2. Если получилось — сообщить пользователю что сделано и какой результат.
3. Если **не получилось** — указать **конкретную причину** (permission denied, такой команды нет, нужен sudo, нужен интерактивный ввод, requires GUI, MCP tool не настроен) и **тогда** попросить пользователя.

## Когда сразу можно просить, не пробуя

- **Физическое взаимодействие** — посмотреть на телефон, нажать кнопку на железе, посмотреть в окно, посмотреть на экран другого устройства.
- **Доступ к чужим аккаунтам / устройствам** — войти в личный кабинет под их логином, открыть их email.
- **Принятие решений и выборов** — это не действие, это согласование. Альтернативы давать, не выполнять самовольно.
- **Команды, которые меняют глобальное состояние без отката** — `rm -rf /`, force-push в main, deploy в prod без явной просьбы. Тут не «не получилось», а «не имею права без явного OK».

## Почему

Просить пользователя сделать `npm install`, `git status`, `ls -la`, открыть файл, который я могу прочитать через Read — это перекладывание моей работы. Пользователь нанял меня, чтобы я делал, а не диктовал ему шаги.

Если действие требует тонкости (например, ввести пароль, который я не знаю) — это валидный случай попросить. Если действие я просто **поленился** или **не подумал** — это failure.
