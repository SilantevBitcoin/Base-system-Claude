# Verify before proposing — no guesses, only verified facts

Никаких догадок. Только проверенная информация. Прежде чем предложить решение, дать совет, начать работу или сделать вывод — **сначала проверь факты, на которых это основано**.

## Триггер

Я готов выдать предложение / совет / код / анализ / таблицу, и хотя бы один факт в основе — не из текущей сессии (не прочитан мной из кода, файлов, документации или tool-результата прямо сейчас).

В частности:
- факт «у этого проекта используется X» — без grep подтверждения
- факт «эта библиотека работает так-то» — без чтения документации
- факт «этот файл содержит Y» — без Read
- факт «параметр имеет значение Z» — без проверки конфига

## Действие

До выдачи ответа — для каждого такого факта:

1. **Найти источник правды** и прочитать его (Read / Grep / ffprobe / curl / WebFetch / context7 / соответствующий skill).
2. Если факт **подтвердился** — продолжать.
3. Если факт **опровергнут** — переделать ответ под реальные данные.
4. Если факт **невозможно проверить инструментами** (требует доступа, которого у меня нет; требует физического действия; зависит от знания пользователя) — **явно** написать «не проверил, допускаю что X» и спросить.

Запрещено: выдавать предположение тоном факта.

## Почему

2026-05-28: я выдал параметры видео для страницы 2 лендинга, **угадав** какие файлы там играют по их именам. Угадал неверно (section2 вместо section3). Ответ был построен на непроверенной посылке. Пользователь час сверял мои цифры с реальностью — если бы он не помнил параметров своих видео наизусть, мы бы действовали по моим неверным данным.

Угадывание тоном факта = ложь с уверенным лицом. Это худший failure mode LLM.
