# Protect dev-system — не менять систему без прямого разрешения владельца

Файлы системы разработки — **read-only по умолчанию**. Claude НЕ изменяет их по своей инициативе;
только когда владелец в текущей сессии явно и конкретно попросил изменить именно систему.

## Триггер

Я собираюсь Write / Edit / создать / удалить / переместить (`rm`/`mv`) файл по одному из путей:
- `<корень системы>/**` — ядро системы (оглавление, маршруты этапов, справочники, рунбуки, runtime-copy, указатель `dev-system.md`).
- `~/.claude/agents/**` — роль-агенты.
- `~/.claude/rules/**` — привязки направлений, дисциплина, поведенческие правила (включая это).
- `~/.claude/skills/**` — скиллы-инструменты.
- `~/.claude/scripts/**` — скрипты хуков.
- `~/.claude/settings.json` — конфиг (плагины, хуки, permissions).

## Действие

1. **Чтение — свободно.** Читать, анализировать, цитировать систему можно всегда.
2. **Запись / правка / удаление — СТОП**, если нет прямого разрешения. Разрешение = владелец в этой
   сессии явно попросил изменить систему («поправь в Base…», «добавь правило…», «обнови агента…»,
   «собери колонку…»).
3. **НЕ считается разрешением:** моя инициатива «улучшу / подчищу / дополню»; косвенный вывод «так
   логичнее»; общая задача проекта, не про систему; то, что я «заметил недочёт». В этих случаях —
   не трогаю, а **предлагаю словами и спрашиваю**.
4. Сомневаюсь, разрешение это или нет → считаю, что **НЕТ**, и спрашиваю.

## Почему

Система — ценный накопленный актив. Правка по инициативе Claude (даже «улучшение») = риск порчи или
тихого дрейфа без ведома владельца. Владелец держит изменения системы под прямым контролем. git откатывает, но
не вносить лишнего дешевле, чем откатывать. Решение владельца — Вариант 1 защиты (инструкция + git);
интерактивный prompt в моменте недоступен, пока включён `bypassPermissions`.
