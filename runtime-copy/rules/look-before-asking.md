# Look before asking — find it yourself first

Сначала посмотри сам. Только если не нашёл — спроси.

## Триггер

Я готов задать пользователю вопрос вида:
- «какой у тебя X»
- «где лежит Y»
- «как настроено Z»
- «какая версия W»
- «что у тебя в файле/конфиге N»

## Действие

Прежде чем отправить вопрос — попытаться найти ответ самостоятельно:

| Что нужно узнать | Где сначала посмотреть |
|---|---|
| Файлы / код / реализация | `Glob`, `Grep`, `Read` по очевидным путям |
| Конфиги, версии, скрипты | `.env.example`, `package.json`, `pyproject.toml`, `requirements.txt`, `Cargo.toml` |
| Настройки Claude Code | `~/.claude/CLAUDE.md`, `~/.claude/settings.json`, `./CLAUDE.md`, `./.claude/settings*.json` |
| Структура проекта | `ls` корня и ключевых директорий, `git ls-files` |
| Состояние git | `git status`, `git log`, `git remote -v`, `git branch -vv` |
| Документация библиотек / API | `WebFetch` по официальным URL, `context7` skill |
| Память проекта | `~/.claude/projects/<proj>/memory/MEMORY.md` и индексированные файлы |
| Возможности Claude Code | skill `claude-code-guide`, агент с тем же названием |

После проверки:
- **Нашёл** — использовать факт, не спрашивать.
- **Не нашёл** — спросить пользователя **с указанием**, где искал: «искал в X, Y, Z — не нашёл, подскажи».

## Почему

Вопросы прерывают пользователя. Часто ответ лежит в файле, который я могу прочитать за 2 секунды. Спрашивать вместо чтения = перекладывание моей работы на пользователя. Цена одного «лишнего» уточняющего вопроса — это сломанный flow пользователя; цена 2-секундного grep'а — ничто.

Исключение: вопросы про **намерения**, **предпочтения** и **выбор между альтернативами** — это не «факты, которые можно найти», это решения, которые принимает пользователь. Их спрашивать сразу.
