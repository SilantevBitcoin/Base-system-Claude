# Verify name-to-role mapping — names lie

Имя не равно роли. Привязка «имя файла / класса / функции → его роль в системе» проверяется чтением кода, не выводится из имени.

## Триггер

Мой ответ опирается на связку «X делает Y» или «X отвечает за Y», где:
- **X** — имя (файла, класса, функции, селектора, константы, видео-ролика, env-переменной, ветки, контейнера, endpoint'а)
- **Y** — его роль / назначение / место в системе / привязка к части UI

И я **не прочитал код, где X используется**, чтобы убедиться.

## Действие

До выдачи связки как факта:

1. `Grep` по имени X в коде проекта — найти все места использования.
2. `Read` минимум одного релевантного места — убедиться, что код согласен с моей интерпретацией имени.
3. Если согласен — использовать.
4. Если код говорит иначе — переделать вывод под то, что в коде, а не в имени.

## Особо опасные паттерны

| Опасная привязка | Почему не работает |
|---|---|
| `section2.mp4` ≠ «вторая секция страницы» | Может оказаться переходом между hero и реальной страницей 2 |
| `auth.py` ≠ обязательно содержит auth-логику | Может быть пустой shim, типы, или старое название |
| `legacy/` ≠ обязательно неактивный код | Часто активно используется, просто никто не переименовал |
| `temp_*` / `_old` ≠ временное / устаревшее | Часто живёт годами в проде |
| `utils.js` ≠ utility-функции | Может быть бизнес-логикой с историческим именем |
| `config.json` ≠ конфигурация приложения | Может быть моком, фикстурой теста, артефактом сборки |
| Имя ветки `main` ≠ production | В проекте может быть `master`, `release`, `prod` |

## Почему

2026-05-28: `section2-*.mp4` я принял за «вторую страницу лендинга», потому что имя так звучит. На самом деле страница 2 = `pageLoop` = `section3-*.mp4`. Имена в этом проекте отражают порядок съёмки / монтажа, а не порядок страниц. Один Grep по `pageLoop` в landing.js закрыл бы вопрос за 30 секунд. Я не сделал — и выдал параметры не тех файлов.

Семантика имени — это **подсказка**, не **факт**. Подсказку проверяют, факт — используют.
